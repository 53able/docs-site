Symphony

Issue tracker の作業を読み、Issue ごとに隔離した workspace で coding agent を走らせる、長時間稼働のスケジューラ / ランナー。

1 — 概要
何をするものか

Symphony は、Linear などの Issue tracker から候補作業をポーリングし、各 Issue に専用 workspace を作って、Codex app-server のような coding agent を継続実行するサービス。エンジニアは個々の agent を逐一見張るのではなく、作業単位とレビュー状態を管理する。

2 — 構成
読み取りから実行まで

仕様上の主要コンポーネントは、設定を読む層、Issue を正規化する層、実行を割り当てる層、workspace と agent process を扱う層、そしてログや dashboard を出す観測層に分かれる。

01Workflow Loader`WORKFLOW.md` の YAML front matter と prompt body を読む。
02Tracker ClientLinear API から active state の Issue を取得し、安定した Issue model に正規化する。
03Orchestratorpoll tick、並行数、retry queue、running state、reconciliation を所有する。
04WorkspaceIssue identifier から安全な path を作り、hooks を実行し、不要 workspace を掃除する。
05Agent Runnerprompt を組み立て、Codex app-server session を開始し、turn を継続する。
Policy

`WORKFLOW.md` がチーム固有の手順、検証、handoff、チケット操作方針を保持する。つまり agent の振る舞いもコードと同じ repo でバージョン管理される。

Coordination

Orchestrator は daemon として tick を回し、候補 Issue、実行中 Issue、retry、terminal state cleanup を調停する。

Execution

Workspace と AgentRunner は、agent のコマンド実行範囲を per-issue workspace に閉じ込め、Codex app-server の turn を進める。

3 — WORKFLOW.md
設定 + プロンプトの契約

`WORKFLOW.md` は YAML front matter と Markdown body でできている。front matter は tracker、polling、workspace、hooks、agent、codex などの実行設定。Markdown body は Issue 情報を埋め込んだ Codex session prompt になる。

WORKFLOW.md minimal shape

  project_slug: "..."

  max_concurrent_agents: 10

---

4 — 実行ループ
daemon としての調停

Orchestrator は `GenServer` として poll tick を持ち、設定を再読み込みしながら候補 Issue の dispatch と実行中 agent の reconciliation を行う。agent process が正常終了しても Issue がまだ active なら continuation retry として再開される。

Dispatch path

  • poll: tracker から active Issue を取得。
  • select: bounded concurrency と claimed / running state を見て対象を選ぶ。
  • spawn: Issue ごとの workspace と Codex session を作る。
  • observe: token totals、rate limits、workspace path、worker host を dashboard へ渡す。

Reconciliation path

  • normal exit: Issue が active なら短い continuation delay で再実行。
  • failure exit: failure retry base delay から retry をスケジュール。
  • terminal state: Done / Closed / Cancelled / Duplicate なら active agent を止め、workspace を掃除。
  • reload failure: workflow reload に失敗したら last known good workflow を維持する。
5 — Elixir 実装
OTP で長時間プロセスを監督する

`elixir/` は現在の参照実装。`mix.exs` では escript `bin/symphony` を生成し、Phoenix LiveView / Bandit / Req / Jason / YamlElixir / Solid などを利用している。README は Elixir を選ぶ理由として、BEAM / OTP の長時間プロセス監督と hot code reloading を挙げている。

6 — 運用と制約
安全に見るべき境界

Symphony の設計は「作業を高い抽象度で管理する」方向に強い。一方で、仕様は強い sandbox や approval posture を一律には要求しない。運用側は trust model、agent tool の権限、workspace root、tracker token、Codex auth を明示する必要がある。

注意: このページは README、SPEC、Elixir 実装ファイルから読み取れる範囲の解説。実運用の安全性は、採用する Codex app-server バージョン、approval policy、host OS sandbox、Linear / GitHub 権限設計に依存する。

チケット状態active / terminal state の読み取りと reconciliation。状態遷移の prompt、agent tool の書き込み権限。
再試行normal exit continuation と failure backoff。最大 turn、failure の許容回数、human review への逃がし方。
観測structured logs、任意の Phoenix dashboard / JSON API。ログ保存先、token accounting、レビュー証跡の扱い。
