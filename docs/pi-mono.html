1 — 概要
何か
pi.dev 0.71.0 MIT TypeScript

pi-mono は、AI エージェントを作るための TypeScript モノレポ。ルート README は「Tools for building AI agents」と定義し、中心となる @mariozechner/pi-coding-agent は minimal terminal coding harness として位置づけられている。

Pi は「ワークフローをツールに合わせる」のではなく、拡張・スキル・プロンプトテンプレート・テーマ・Pi Packages で ユーザー側の運用に合わせて変形する ことを狙っている。

2 — Pi coding agent
使い始め

@mariozechner/pi-coding-agent は、read・write・edit・bash を標準ツールとして持つ対話型コーディングエージェント CLI。API キーでも、既存サブスクリプションの /login でも始められる。

npm install -g @mariozechner/pi-coding-agent

# API key で起動
export ANTHROPIC_API_KEY=sk-ant-...
pi

# 既存サブスクリプションで認証
pi
/login
interactive

標準 TUI。エディタ、メッセージ、ツール呼び出し、フッターで状態を見ながら作業する。

-p

一回だけ応答して終了。標準入力も初期プロンプトに混ぜられる。

--mode json

イベントを JSON Lines で出す。ログ処理や非対話実行向け。

--mode rpc

stdin/stdout の LF 区切り JSONL RPC。Node 以外のプロセス連携向け。

3 — パッケージ一覧
モノレポ構成

全主要パッケージは 0.71.0 で揃っている。Pi 本体だけでなく、統一 LLM API、低レベルのエージェントランタイム、差分レンダリング TUI、AI チャット向け Web UI に責務が分かれている。

パッケージ 役割
@mariozechner/pi-coding-agent Pi CLI 本体。read / bash / edit / write ツール、セッション管理、TUI、JSON / RPC / SDK 連携を提供。
@mariozechner/pi-ai 自動モデル発見、プロバイダー設定、トークン・コスト計測、コンテキスト永続化、モデル間ハンドオフを持つ統一 LLM API。
@mariozechner/pi-agent-core ツール呼び出し、状態管理、添付ファイル対応を持つ汎用エージェント基盤。
@mariozechner/pi-tui 差分レンダリング付きターミナル UI ライブラリ。
@mariozechner/pi-web-ui @mariozechner/pi-ai を使う AI チャット向け再利用 Web UI コンポーネント。
4 — セッションと共有
JSONL + tree

Pi のセッションは JSONL ファイルとして保存され、各エントリが id と parentId を持つ。これにより、別ファイルを増やさずに履歴ツリー上で分岐できる。

/tree

過去の任意地点へ移動し、そこから続行。検索、折りたたみ、ラベル付けもできる。

/fork

過去のユーザーメッセージから新しいセッションファイルを作る。

/clone

現在のアクティブブランチを新しいセッションに複製する。

/compact

古いメッセージを要約し、長いセッションのコンテキスト逼迫を回復する。

pi -c                  # 最新セッションを継続
pi -r                  # 過去セッションを選択
pi --no-session        # 保存しない一時セッション
pi --session <path|id> # 特定セッションを利用
pi --fork <path|id>    # 特定セッションから分岐
OSS セッション共有

README は、OSS で Pi や他の coding agent を使った作業セッションの共有を呼びかけている。狙いは、toy benchmark ではなく実作業のツール呼び出し・失敗・修正を集め、モデル・プロンプト・ツール・評価を改善すること。

共有ツールは badlogic/pi-share-hf。作者自身の作業セッションは badlogicgames/pi-mono として Hugging Face で公開されている。

5 — 拡張モデル
core stays small

Pi はサブエージェントや plan mode を標準装備しない。その代わり、TypeScript 拡張と Agent Skills 標準、プロンプトテンプレート、テーマ、Pi Packages で、各チームのワークフローを後付けする。MCP、権限ゲート、サンドボックス、Git チェックポイント、独自 UI も拡張側で組み込める。

Prompt Templates

Markdown の再利用プロンプト。/name で展開し、変数も使える。

Skills

Agent Skills 標準の手順パッケージ。/skill:name または自動ロードで使う。

Extensions

TypeScript でツール、コマンド、キー、イベント、UI コンポーネント、プロバイダーを追加する。

Pi Packages

extensions / skills / prompts / themes を npm または git で配布・インストールする。

セキュリティ注意: Pi Packages はシステム権限で動く。拡張は任意コードを実行でき、スキルはモデルに任意の操作を指示できるため、第三者パッケージはソース確認が前提。
インストール例 
pi install npm:@foo/pi-tools
pi install npm:@foo/pi-tools@1.2.3
pi install git:github.com/user/repo
pi install https://github.com/user/repo@v1
pi remove npm:@foo/pi-tools
pi list
pi update
pi config
pi update --extensions
pi update --self
pi update npm:@foo/pi-tools
6 — 開発・貢献
ビルド・検証

リポジトリルートで以下を実行する。npm run check は npm run build の後に実行する必要がある。web-ui が依存パッケージのコンパイル済み .d.ts を必要とするため。

npm install          # 依存関係のインストール
npm run build        # 全パッケージのビルド
npm run check        # Biome / tsgo / browser smoke / web-ui check
./test.sh            # API キーなしなら LLM 依存テストはスキップ
./pi-test.sh         # ソースから pi を実行
Contribution gate

新規 contributor の Issue / PR はデフォルトで自動クローズされる。メンテナーが日次で見直し、質の高いものだけを reopen または承認する。

  1. 1短く具体的で、1 画面に収まる Issue を出す。バグや要望と「なぜ重要か」を明確にする。
  2. 2lgtmi なら今後の Issue、lgtm なら今後の Issue と PR が承認済み扱いになる。
  3. 3PR 前に npm run check と ./test.sh を通す。CHANGELOG.md は直接編集しない。
唯一のルール: 自分のコードを理解していること。AI で書くのは問題ないが、変更内容と影響を説明できない PR はクローズされる。
