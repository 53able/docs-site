GSD-2

Get Shit Done 2 — 自律型コーディングエージェント CLI（npm: gsd-pi）

1 — 概要
何か

GSD-2 は TypeScript で書かれた自律型コーディングエージェント CLI。
gsd auto と打つだけで、LLM が 設計 → 研究 → 計画 → 実装 → 検証 → コミット まで全工程を自律実行する。 v1（Claudeへのプロンプト注入フレームワーク）の進化版で、Pi SDK 上の TypeScript アプリとして LLM セッションを直接制御する。 ディスクドリブンな状態機械 (.gsd/) により、クラッシュ後もどこからでも再開可能。

セッション設計の哲学

「1 タスク = 1 新鮮セッション」がコアの思想。タスクをまたいでコンテキストを引き継がないことで、200k トークンウィンドウを毎回クリーンに使い切る。.gsd/ フォルダ（STATE.md, ROADMAP, PLANS）がセッション間の唯一の真実として機能する。

トークン削減: token_profile: budget を preferences に設定すると、プロンプト圧縮により 40〜60% のコスト削減が可能（v2.17 以降）。
4 — ワーク階層
Milestone → Slice → Task

GSD-2 はすべての作業を 3 層の階層で管理する。各層は「1 コンテキストウィンドウに収まる」単位が設計指針。

Milestone

出荷可能なバージョン単位

4〜10 スライスで構成。Git worktree を 1 本割り当て、完了時に squash merge される。例: M001, M002

1 マイルストーン = 4〜10 スライス
Slice

デモ可能な機能単位

1〜7 タスクで構成。スライスが完了するたびにデモ可能な状態になる。例: S01, S02

1 スライス = 1〜7 タスク
Task

1 コンテキストウィンドウ単位

1 セッション（200k トークン）に収まる最小作業単位。LLM が 1 回のセッションで完結させる。例: T01

Git 戦略 
main:
  feat(M001/S02): API実装      ← squash merge（マイルストーン完了時）
  feat(M001/S01): データモデル

milestone/M001（worktree 内）:
  feat(S01/T03): ファイルライター実装
  feat(S01/T02): パーサー実装
  feat(S01/T01): コアタイプ定義
