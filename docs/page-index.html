PageIndex

Vectorless · Reasoning-based RAG — 長文ドキュメントを階層ツリーで索引し、推論でたどり着く検索

1 — 概要
何を解くか

従来のベクトル RAG は意味的類似度で近いチャンクを返すが、専門文書では「似ていること」と「答えに必要なこと」が一致しないことが多い。PageIndexは、長文 PDF などから目次に相当する階層ツリーを生成し、LLM がその索引の上でツリー探索（推論）して関連セクションへ辿る、ベクトル DB 不要の RAG パイプラインだ。

2 — 処理の流れ
2 ステップ

README が示すコアは次の二段だ。

1. ツリー生成 → ドキュメントの ToC 風・階層構造 JSON → 2. ツリー探索 → 推論に基づくセクション選択・取得
  • 類似度検索ではなく経路探索: 人が章立てを辿るように、LLM がノードの要約や見出しを手がかりに枝を選ぶイメージ。
  • 説明可能性: どの節を読んだかが追いやすく、ページ・セクション参照とセットで語れる。
3 — ツリー索引
JSON のイメージ

各ノードにはタイトル・ノード ID・ページ範囲・要約・子ノードが載る。長大な規制文書や決算資料など、コンテキスト上限を超える資料向けに設計されている。

{
  "title": "Financial Stability",
  "node_id": "0006",
  "start_index": 21,
  "end_index": 22,
  "summary": "The Federal Reserve ...",
  "nodes": [
    {
      "title": "Monitoring Financial Vulnerabilities",
      "node_id": "0007",
      "start_index": 22,
      "end_index": 28,
      "summary": "The Federal Reserve's monitoring ..."
    }
  ]
}
Markdown 入力: run_pageindex.py --md_path では見出しの # 階層でノードを切る。PDF から機械変換した MD では階層が崩れやすいので、その場合は PDF からの直接パイプラインやクラウド側の高度な変換を検討する、と README で注意されている。
4 — ローカル利用（OSS）
最短パス

リポジトリ直下で依存を入れ、LiteLLM 対応の API キーを用意してから PDF を渡す。

pip3 install --upgrade -r requirements.txt

ルートに .env を置き（例）:

OPENAI_API_KEY=your_openai_key_here
python3 run_pageindex.py --pdf_path /path/to/your/document.pdf

主な CLI オプション（抜粋）

フラグ意味・既定の例
--model利用 LLM（既定は README 掲載の GPT-4o 系）
--toc-check-pages目次探索に見る先頭ページ数（既定 20）
--max-pages-per-nodeノードあたり最大ページ数（既定 10）
--max-tokens-per-nodeノードあたり最大トークン（既定 20000）
--if-add-node-idノード ID を付与するか（yes/no）
--if-add-node-summaryノード要約を付与するか（yes/no）
--md_pathPDF の代わりに Markdown からツリー生成

エージェント連携のサンプルとして examples/agentic_vectorless_rag_demo.py（OpenAI Agents SDK）や cookbook/ 配下の Notebook が用意されている。
