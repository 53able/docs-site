Compozy（compozy/compozy）

Go 単一バイナリ · Markdown 成果物 · ACP ランタイム — アイデアから出荷までのパイプライン CLI

1 — 概要
位置づけ

Compozy は、散在するプロンプト・手作業のタスク管理・コピペレビューを一つの CLI にまとめるツール。プロダクト構想から PRD・技術仕様・タスク分解・並列エージェント実行・レビュー修正までを、読み書き可能な Markdown とローカルデーモンでつなぐ。ランタイム依存はなく、コードとデータはマシン上に残る（README の説明に基づく）。

Go単一バイナリ
40+setup 対象エージェント
ACP実行トランスポート
MD成果物は Markdown

リポジトリはモノレポ構成。CLI（cmd/compozy）、組み込みスキル（skills/）、Web UI（web/）、拡張 SDK（sdk/）、OpenAPI 定義（openapi/）が同居する。

2 — ワークフロー（Idea → Ship）
フェーズ

README の Quick Start に沿った標準パイプライン。各段階は次段の入力になる Markdown を残す。スコープが既にある場合は Idea フェーズを省略できる。

Idea（任意） → PRD → TechSpec → Tasks → 実行 → Review → Fix
段階入口主な出力
アイデア（任意）/cy-idea-factory（拡張）調査付きアイデア仕様
PRD/cy-create-prdビジネス PRD + ADR
技術仕様/cy-create-techspecアーキテクチャ・API 設計
タスク分解/cy-create-taskstask_*.md（v2 frontmatter）
実行compozy tasks run実装 + ステータス更新
レビュー/cy-review-round または reviews fetchreviews-NNN/issue_*.md
修正compozy reviews fix修正 + プロバイダスレッド解決
タスク v2: frontmatter は status · title · type · complexity · dependencies。実行前は compozy tasks validate --name <feature> で検証できる。
3 — デーモンランタイム
役割分担

ワークスペース内の成果物（.compozy/tasks/<name>/）と、ホーム配下のランタイム状態（~/.compozy/）を分離する。タスク実行・レビュー修正・永続 exec はデーモンがスナップショットとストリームを提供する。

flowchart TB
  subgraph ws["ワークスペース .compozy/tasks/"]
    PRD[PRD / TechSpec]
    TASK[task_*.md]
    REV[reviews-NNN/]
    MEM[memory/]
  end
  subgraph home["~/.compozy/"]
    DM[daemon]
    RUNS[runs/run-id/]
    DB[global.db]
  end
  CLI[compozy CLI] --> DM
  DM --> RUNS
  CLI --> ws
  sync[compozy sync] --> DB
  ws --> sync

主要コマンド

  • compozy daemon start|status|stop — ホームスコープのデーモン生命周期
  • compozy workspaces list|register|resolve — ワークスペース登録表
  • compozy tasks run <slug> — 既定は TUI アタッチ。非対話ではストリーム
  • compozy runs attach|watch <run-id> — 既存ランの再開・観察
  • compozy sync — 成果物を global.db に同期。レガシー _meta.md の整理も含む
マイグレーション: 旧 XML タグ付き成果物がある場合は、デーモン連携コマンドの前に一度 compozy migrate を実行する（README）。
4 — 成果物と設定
2 層のストレージ
場所内容編集
.compozy/tasks/<name>/PRD、TechSpec、タスク、レビュー、memory人間が Git で diff 可能
~/.compozy/runs/<run-id>/実行 DB、events.jsonl、ターンごとの prompt/responseデーモン管理。attach で観察
~/.compozy/config.tomlグローバル既定（ide、model、timeout 等）任意
.compozy/config.tomlワークスペース上書き任意。CLI フラグが最優先

設定の優先順位（README）:

explicit flags > workspace [command] > workspace [defaults]
  > global [command] > global [defaults] > built-in defaults

再利用可能エージェント

.compozy/agents/<name>/（ワークスペース）と ~/.compozy/agents/<name>/（グローバル）。必須 AGENT.md、任意 mcp.json。同名はワークスペース側が丸ごと勝つ。

compozy agents list
compozy agents inspect reviewer
compozy exec --agent reviewer "Review the staged changes"

Workflow Memory

memory/MEMORY.md（ワークフロー横断）と memory/task_NN.md（タスク単位）の 2 層。実行前に読み込み、完了前に更新。行数・サイズのソフトリミット超過時はコンパクションを促す（cy-workflow-memory スキルがプロンプトに組み込まれる）。

コア 8 + 任意拡張

compozy setup で Claude Code、Codex、Cursor など検出済みエージェントへコアスキルをシンボリックリンク（既定）またはコピーで配布する。

6 — 拡張（Extensions）
サブプロセスプラグイン

バイナリを組み替えず、JSON-RPC 2.0（stdin/stdout）でパイプライン各フェーズにフックする。TypeScript（@compozy/extension-sdk）と Go（sdk/extension）の SDK、npx @compozy/create-extension でテンプレート生成。

compozy ext list
compozy ext install compozy/compozy --remote github --ref <tag> --subdir extensions/cy-idea-factory
compozy ext enable cy-idea-factory
compozy ext doctor

テンプレート種別: lifecycle-observer · prompt-decorator · review-provider · skill-pack。発見スコープは workspace > user > bundled。user/workspace 由来は明示的 enable が必要。

Ad hoc 実行

ワークフローを張らず 1 プロンプトだけ走らせる compozy exec。出力は --format text|json|raw-json。--persist で ~/.compozy/runs/<run-id>/ に再開可能なセッションを保存する。

7 — 導入と最小実行
インストール 
# Homebrew
brew tap compozy/compozy && brew install --cask compozy

# npm
npm install -g @compozy/cli

# Go
go install github.com/compozy/compozy/cmd/compozy@latest

# スキル配布
compozy setup

機能デモ用のエージェントループ画面（README スクリーンショット）:

例: user-auth ワークフロー

  1. compozy setup
  2. エージェント内で /cy-create-prd user-auth → /cy-create-techspec → /cy-create-tasks
  3. compozy tasks run user-auth --ide claude
  4. compozy reviews fetch user-auth --provider coderabbit --pr 42 または /cy-review-round
  5. compozy reviews fix user-auth --ide claude --concurrent 2
