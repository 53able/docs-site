3 — 開発コマンド
日次で触るもの

README.md と DEVELOPMENT.md に揃えた要約。詳細オプションは upstream のドキュメントを参照。

コマンド内容
bun install初回セットアップ。
bun startローカルページサーバ（既定でポート 3000 付近）。/demos/index などを開く。
bun run check型検査 + Lint。
bun test小さめの不変条件テスト。
bun run build:packagenpm 向け dist/ を生成。
bun run site:build静的デモサイトを site/ にビルド。
bun run generate:bidi-data簡略化 Unicode bidi レンジの再生成。

ブラウザ検証系（抜粋）

shell
bun run accuracy-check
bun run benchmark-check
bun run corpus-sweep
bun run corpus-status

ブラウザターゲット別に :safari / :firefox サフィックスが付くタスクが並ぶ。スナップショット更新は accuracy-snapshot 系や letter-spacing-snapshot など名前に snapshot が付いたスクリプトで行う。

4 — 精度・ベンチマーク
チェックイン JSON

STATUS.md によれば、精度スイープは「フォント × サイズ × 幅 × テキスト集合」の行列をブラウザで走らせた結果を JSON に固定している。公開の /accuracy ページは回帰ゲート寄りで、日々の意思決定の主指標ではない、という位置づけ。

ベンチ: チェックイン値は 3 回冷起動チェックの中央値。ローカル単発のページ数値とはズレうるため、ホットパス変更時は src/analysis.ts / measurement.ts / line-break.ts / layout.ts / bidi.ts または pages/benchmark.ts を触ったらスナップショット更新手順を踏む、と upstream が案内している。
5 — コーパス
長文とスイープ

corpora/ には言語横断のテキストファイルと、Chrome/Safari の step=10 幅スイープ結果、dashboard.json などが置かれる。corpus-check で単一コーパスを狭い幅集合で診断し、corpus-taxonomy で不一致フィールドをバケツ分けする、といった運用が DEVELOPMENT.md に列挙されている。
