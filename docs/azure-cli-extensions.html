Azure CLI Extensions

azure-cli-extensions — マルチ拡張機能モノリポ（公式 Azure CLI 拡張エコシステム）

1 — 概要
リポジトリの目的

azure-cli-extensions は Microsoft 公式の Azure CLI 拡張機能モノリポジトリ。 2 つの独立した目的を持つ：① 拡張機能ソースコードのホスト（src/ ディレクトリ）、 ② 公開インデックス（src/index.json）への登録によるグローバルな配信。

208公開拡張機能数
Python3.6+ 対応
azdev開発ツールチェーン
数分index.json 同期間隔

インデックスは https://aka.ms/azure-cli-extension-index-v1 へ数分おきに同期され、 az extension add --name NAME による名前指定インストールが可能になる。 ソースコードをこのリポジトリに置く必要はなく、外部リポジトリにホストして index.json のみ登録することも可能。

インデックス経由のインストール 
# 名前指定インストール（index.json 登録済み）
az extension add --name NAME

# 利用可能な拡張機能一覧
az extension list-available

# 更新
az extension update --name NAME
外部ホスト（非インデックス） 
# URL 指定でインストール
az extension add \
  --source https://contoso.com/myext.whl

# ※ 名前指定・list-available 不可
# ※ 更新は remove → add で手動対応

外部ホストでもソースコードは本リポジトリに置ける。インデックス登録だけ任意。

3 — 拡張機能インデックス（index.json）
index.json の役割

src/index.json はリポジトリの「公開ディレクトリ」。 マージされた PR から自動パイプラインが WHL をビルドしてアップロードした後、 index.json の更新 PR を自動作成する仕組みになっている。 開発者は setup.py のバージョンを上げるだけで、index.json を手動で編集する必要はない。

注意： index.json に登録された拡張機能はすべて公開になり、全ユーザーが利用可能になる。 プライベートな配布には外部ホスト方式を使うこと。
エントリ構造 
{
  "extensions": {
    "aks-preview": [           // 拡張機能名（キー）
      {
        "downloadUrl": "https://azcliprod.blob.core.windows.net/...",
        "filename": "aks_preview-20.0.0b1-py3-none-any.whl",
        "metadata": {
          "azext.isPreview": true,
          "azext.minCliCoreVersion": "2.70.0",
          "version": "20.0.0b1",
          "summary": "Provides a preview for upcoming AKS features",
          "classifiers": ["Development Status :: 4 - Beta", ...]
        },
        "sha256Digest": "abc123..."   // 小文字で記載必須
      }
    ]
  },
  "formatVersion": "1"
}
SHA256 の生成： azdev extension update-index <URL> を使うと自動計算される。 手動では macOS: shasum -a 256 path.whl、Windows: Get-FileHash path.whl -Algorithm SHA256。 ハッシュは小文字必須（大文字だと CI 失敗）。
主要拡張機能サンプル
拡張機能名 概要 ステータス
aks-preview AKS の Preview 機能を先行提供。azure-cli の aks コマンドをオーバーライド Preview
spring Azure Spring Apps の管理コマンド GA

4 — 拡張機能の構成（アナトミー）
標準ディレクトリ構造 
src/<extension-name>/
├── setup.py              # パッケージメタデータ・バージョン定義
├── setup.cfg             # 追加パッケージ設定
├── README.rst            # 拡張機能の説明・使い方
├── HISTORY.rst           # 変更履歴（バージョンごと）
├── linter_exclusions.yml # リンター個別除外設定
└── azext_<name>/         # メインパッケージ（ハイフン→アンダースコア）
    ├── __init__.py       # コマンドローダー登録
    ├── _params.py        # CLI パラメータ定義
    ├── _help.py          # ヘルプテキスト
    ├── _validators.py    # 入力バリデーター
    ├── commands.py       # コマンド登録（load_command_table）
    ├── custom.py         # カスタムコマンド実装
    ├── _client_factory.py # SDK クライアント生成
    └── vendored_sdks/    # AutoRest 生成 SDK（静的解析除外）
vendored_sdks について： AutoRest で生成した SDK は CI の静的解析（flake8/pylint）を通過しないことが多い。 azext_*/vendored_sdks/ に配置することで自動的に静的解析から除外される。
setup.py の重要フィールド 
from setuptools import find_packages, setup

VERSION = "20.0.0b1"   # ← バージョンはここだけ更新（index.json は触らない）

CLASSIFIERS = [
    "Development Status :: 4 - Beta",   # Preview の場合
    # "Development Status :: 5 - Production/Stable",  # GA の場合
    "Programming Language :: Python :: 3",
]

setup(
    name="aks-preview",
    version=VERSION,
    description="Provides a preview for upcoming AKS features",
    long_description=README,
    classifiers=CLASSIFIERS,
    packages=find_packages(),
    install_requires=DEPENDENCIES,
)
azext_metadata.json（任意）

拡張機能ルートに azext_metadata.json を置くと追加メタデータを定義できる。 azext.isPreview・azext.minCliCoreVersion・azext.isExperimental などを指定可能。 index.json のエントリと一致していないと CI が失敗する。

5 — 開発フロー（azdev）
azdev — 開発ツールチェーン

azure-cli-dev-tools（azdev） が スタイルチェック・リンター・テスト・インデックス更新をまとめて提供する。 開発コンテナ（.devcontainer/）を使えば azdev が自動セットアップされる。

Step 01
環境セットアップ
pip install azdev
または Dev Container を開く
Step 02
スタイルチェック
azdev style <YOUR_EXT>
flake8 + pylint でコードスタイル検証
Step 03
リンター実行
azdev linter <YOUR_EXT>
コマンド設計ルール・命名規約の検証
Step 04
インデックス検証
python scripts/ci/test_index.py -q
(pip install wheel==0.30.0 必要)
Step 05
バージョン更新
setup.py の VERSION を更新
HISTORY.rst に変更内容を記載
Step 06
index 更新（任意）
azdev extension update-index <URL>
外部ホスト時に手動で index.json を更新
Git フック

.githooks/ に pre-commit・pre-push フックが用意されている。 azdev_active.sh（macOS/Linux）または azdev_active.ps1（Windows）で有効化できる。 フックは azdev がアクティブな環境でのみ動作する。

6 — CI/CD とリリース
自動リリースパイプライン

main ブランチへのマージをトリガーに以下が自動実行される：

PR マージ（main） → WHL ビルド → Azure Blob へアップロード → index.json 更新 PR 自動作成 → index 更新 PR マージ → aka.ms URL へ同期（数分）
開発者の作業範囲： setup.py のバージョン更新と HISTORY.rst の記載のみ。 src/index.json を 直接編集してはいけない（CI 失敗の原因になる）。
Azure Pipelines チェック
CredScan PolicyCheck BreakingChangeTests SyncAlias
  • CredScan: 認証情報・シークレットの混入チェック
  • PolicyCheck: Microsoft ポリシー準拠チェック
  • BreakingChangeTests: 後方互換性の検証
GitHub Actions ワークフロー（13）
AzdevLinter AzdevStyle TriggerExtensionRelease CreateReleaseTag BlockPRMerge CCOA AddPRComment ProcessCodeReview

リンター・スタイル検証は PR 単位で自動実行。リリーストリガーはマージ後に実行。 PR オープン/マージ時の自動コメント通知も含む。

バージョニング規約
ステータスバージョン形式Development Status Classifier
Preview X.Y.ZbN（例: 20.0.0b1） Development Status :: 4 - Beta
GA X.Y.Z（例: 2.8.1） Development Status :: 5 - Production/Stable

詳細は Extension version schema を参照。

7 — コントリビューション
PR チェックリスト
  • 1 azdev style <YOUR_EXT> をローカルで実行済み（pip install azdev 必要）
  • 2 python scripts/ci/test_index.py -q をローカルで実行済み（pip install wheel==0.30.0 必要）
  • 3 バージョンが Extension version schema に準拠している
  • 4 新規拡張の場合: description/summary が Summary Guidelines に準拠している
  • 5 .github/CODEOWNERS に拡張機能のオーナーを追加した
  • 6 src/index.json を直接編集していない（自動更新されるため）
CLA（貢献者ライセンス合意）

初めて PR を出すと CLA ボットが自動でコメントし、 cla.microsoft.com での署名を促す。 一度署名すれば同じ CLA を使う全リポジトリで有効になる。 Microsoft Open Source Code of Conduct に従う。

ソースコードを別リポジトリでホストする場合

ソースコードを外部リポジトリに置いたまま index.json のみ登録できる。 その場合は WHL を PyPI または Azure Blob などにアップロードし、 azdev extension update-index <URL> でエントリを生成して PR を出す。
