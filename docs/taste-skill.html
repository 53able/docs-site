Taste Skill

The Anti-Slop Frontend Framework for AI Agents - implementation skills と image-generation skills を同梱する配布リポジトリ。

1 - 概要
Positioning

Taste Skill は、AI が量産しがちな「それっぽいけど薄い UI」を避けるための anti-slop frontend framework。いまの主語は単なる 4 本の `SKILL.md` ではなく、実装スキル + 画像生成スキル + install 名付きの配布パッケージ に広がっている。

リポジトリの中心は `taste-skill` だけではない。`redesign-skill`、`gpt-tasteskill`、`image-to-code-skill` に加えて、`imagegen-frontend-web`、`imagegen-frontend-mobile`、`brandkit` まで揃い、ChatGPT Images や Codex も含む複数エージェントの前段設計に使える。

2 - Install
CLI first

README の導線は、`SKILL.md` を手でコピーする方法よりも npx skills add を使った導入が中心になっている。リポジトリ全体を入れるか、frontmatter の `name:` を指定して 1 本だけ入れる。

npx skills add https://github.com/Leonxlnx/taste-skill

npx skills add https://github.com/Leonxlnx/taste-skill --skill "design-taste-frontend"

--skill に渡すのはフォルダ名ではなく install name。`taste-skill/` をそのまま指定するのではなく、`name: design-taste-frontend` の値を使う。

補助導線: `SKILL.md` をプロジェクトにコピーしたり、会話に貼り付けたりする使い方も残っている。ただし現行リポジトリは installable な skill catalog として読む方が自然。

4 - v2 default
何が変わったか
  • Brief Inference: いきなり見た目を決めず、ページ種別・対象読者・vibe を 1 行で読む。
  • Design System Map: Material、Fluent、Carbon、Primer、GOV.UK、shadcn などに寄せる条件を明文化。
  • Redesign Protocol: greenfield と preserve / overhaul を分け、監査から入る。
  • Dark Mode Protocol: dual-mode 前提でトークン戦略と contrast を扱う。
  • Final Pre-Flight Check: 出荷前の hard checklist を持つ。
v2 の強化点

今回の更新でいちばん大きいのは、`taste-skill` が「3 つの数字を回すスキル」から、「設計判断の手順と禁止パターンを持つ skill」へ変わったこと。とくに em-dash 完全禁止、section-number eyebrow 禁止、fake screenshot 禁止、`motion/react` 推奨、Tailwind v4 既定は v2 の象徴になっている。

Dial意味
DESIGN_VARIANCEレイアウトの非対称さや実験度を決める。
MOTION_INTENSITYhover から scroll choreography まで motion の強さを決める。
VISUAL_DENSITY1 画面に載せる情報量を調整する。
v1 に pin するとき

v2 は既定だが experimental。旧 behavior をそのまま使いたいときは、README と CHANGELOG にある通り `taste-skill-v1` を install name で pin する。

npx skills add https://github.com/Leonxlnx/taste-skill --skill "design-taste-frontend-v1"
5 - Scope
向いているもの
  • landing pages
  • portfolios
  • redesigns
  • 画像から方向性を固めてから実装したい案件

README とメイン skill の両方が、Taste Skill を「見た目の設計」と「anti-slop な前段づくり」に寄せている。

対象外
  • dashboards
  • data tables
  • multi-step forms / wizards
  • code editors
  • native mobile UI そのもの
  • realtime collaboration UI

つまり「フロントエンド全般を何でもよくする万能 skill」として読むより、マーケ、ポートフォリオ、リデザインの見た目を引き上げる系統として捉える方が実態に近い。

どれを使うか
  • まずは `taste-skill`: 汎用の既定。今は v2 experimental。
  • 互換性優先なら `taste-skill-v1`: 既存運用を崩したくないとき。
  • 既存 UI を直すなら `redesign-skill`: 作り直し前提ではなく audit-first。
  • 画像から始めるなら `imagegen-*` + `image-to-code`: 先に参照フレームを作りたいとき。
  • 方向性を固定したいなら variant skills: `soft-skill`、`minimalist-skill`、`brutalist-skill` を追加する。
  • 出力が途中で切れるなら `output-skill`: placeholder comment や省略コード対策。
