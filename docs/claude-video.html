claude-video

bradautomates/claude-video — Agent Skill /watch · yt-dlp + ffmpeg + captions/Whisper · v0.2.0

1 — 概要
何をするか

エージェントに動画入力を足すスキル。URL（yt-dlp が扱えるサイト）またはローカルパス（.mp4 / .mov / .mkv / .webm）を渡し、キャプションまたは Whisper のタイムスタンプ付き文字起こしと、JPEG フレームを揃えてから Read で画像を読む。

ゼロ設定で開始: 初回に setup.py が ffmpeg / yt-dlp を確認する。macOS は Homebrew で自動インストール。公開動画の多くはネイティブキャプションだけで足り、Whisper API キーはキャプション無しのときだけ必要。

典型ユース

  • 他人の動画のフック・構成分析
  • 画面録画からのバグ診断
  • 長尺の要約・ハイプ除去
  • プレイリストをノート化

製品の境界

  • 本体は CLI ではなくスキル契約 SKILL.md
  • scripts/watch.py は実装エントリ
  • 自己完結フォルダ skills/watch/ を単位で配布
  • ハーネス固有の CLAUDE_SKILL_DIR に依存しない
2 — 処理の流れ
ランタイム

ユーザーがソースと質問を渡すと、エージェントは preflight → watch.py → 全フレーム Read → 回答 → 作業ディレクトリ掃除、の順で動く。

setup --check → download / captions → frames + dedup → transcript → Read frames → answer + cleanup 
flowchart TD
  A["/watch source + question"] --> B["setup.py preflight"]
  B --> C{"captions?"}
  C -->|yes + transcript detail| D["report without video download"]
  C -->|need pixels / no captions| E["yt-dlp download"]
  E --> F["ffmpeg frames"]
  F --> G["dedup near-duplicates"]
  C -->|no captions| H["Whisper audio"]
  D --> I["timestamped transcript"]
  G --> I
  H --> I
  I --> J["Claude Read each JPEG"]
  J --> K["grounded answer"]
  K --> L["rm work dir if done"]
  1. キャプション優先: transcript detail ではキャプション付き URL なら動画ダウンロードをスキップできる。
  2. フレーム抽出: efficient はキーフレーム、balanced / token-burner はシーン変化（不足時は均一サンプリング）。既定幅 512px、高さ上限 1998px。
  3. 文字起こし: ネイティブ字幕 → 失敗時は mono 16 kHz 64 kbps mp3 を Whisper（Groq 優先、OpenAI 代替）。
  4. 回答: フレームパスに t=MM:SS、トランスクリプトにタイムスタンプ。全フレームを並列 Read。
3 — Detail モード
速度とトークンのダイヤル

--detail または ~/.config/watch/.env の WATCH_DETAIL。既定は balanced。測定例は 49:08 / 1280×720 の画面録画（英語 auto-captions）に対する実測。

ModeEngineCap抽出時間（例）画像トークン目安
transcriptなし（captions）—~4.5 s（yt-dlp 1 回）0（テキストのみ）
efficientkeyframe50~0.5 s~9.8k
balancedscene-change100~20.9 s~19.7k
token-burnerscene-changeuncapped~21.0 s~22.8k（例では 116 枚）
  • 512px 幅の 720p フレームは約 197 tokens/frame（Anthropic の (w×h)/750）。
  • 候補を全範囲から取り、even-sample（先頭・末尾は必ず残す）で cap まで間引く。モード差は候補ソースと cap だけ。
  • efficient は「速い」意味。低モーションではキーフレーム数がシーンカットより多くなり得る。
4 — フレーム予算と dedup
トークンはフレームが支配
尺フル動画の目安備考
≤30 s~30 frames密
30 s – 1 min~40密
1 – 3 min~60余裕あり
3 – 10 min~80疎め
> 10 mincap まで疎sparse scan 警告 → --start/--end 推奨

フォーカスモード

ユーザーが時刻を指定したら --start / --end。秒あたり予算が上がり、上限は 2 fps。長尺の全体スキャンより有用。

近傍重複の除去

既定で dedup が走る（--no-dedup で無効）。各 JPEG を 16×16 グレースケールにし、直前に残したフレームとの mean absolute difference ≤ 2.0 なら落とす。cap は dedup 後に適用。

transcript-cue frames: プレゼンターの「look here」などは視覚変化が小さい。トランスクリプトを読んで --timestamps T1,T2,... を足す。cue は cap に対して先に予約され、even-sample で落ちない。

初回セットアップ

  • macOS: brew install ffmpeg yt-dlp を自動実行
  • Linux / Windows: 表示された apt / dnf / winget 等を実行
  • キー: ~/.config/watch/.env（mode 0600）に GROQ_API_KEY（優先）または OPENAI_API_KEY
  • キー無しでも完了可。キャプション無し動画は --no-whisper でフレームのみ

Flag役割

--no-whisper文字起こしフォールバック無効

--out-dir DIR作業ディレクトリ指定

8 — 制約・権限
精度とセキュリティ
  • cap 付きモード（efficient / balanced）は ~10 分超で疎になる。区間再実行か token-burner。
  • ユニバーサル上限 2 fps。250 フレーム超で soft warning。
  • 動画本体は API に送らない。送るのはキャプション欠落時の抽出音声のみ。
  • yt-dlp は公開データのみ。ログイン・Cookie・投稿はしない。
  • キーは stdout/stderr/成果物に出さない。永続は作業 dir と ~/.config/watch/.env のみ。
コスト目安: 80 枚 @ 512px でおおよそ 50–80k image tokens。1024px は約 4 倍。同一セッションの追質問ではスクリプトを再実行しない。
