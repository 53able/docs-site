Trellis

mindfold-ai / npm @mindfoldhq/trellis — specs, tasks, workflow, session memory in-repo

1 — 概要
何か
@mindfoldhq/trellis 15 platforms AGPL-3.0 Node 18+ / Python 3.9+

Trellis は、AI が毎セッションゼロから始まる問題に対し、spec・タスク・ワークフロー・記憶をリポジトリに永続化する out-of-the-box のエンジニアリングフレームワーク。巨大な CLAUDE.md を、必要なときだけ注入される進行型レイヤーへ分割する。

README では、チーム標準を .trellis/spec/ に書き、PRD と状態を .trellis/tasks/ に置き、前回の文脈を .trellis/workspace/ のジャーナルで引き継ぐ、という 5 つの能力（自動 spec 注入・タスク中心・プロジェクト記憶・チーム共有・マルチプラットフォーム）を列挙している。

2 — .trellis/ のレイヤー
ディレクトリの役割

インストール後、リポジトリ直下に .trellis/ が置かれ、各サブツリーがエージェントが参照する知識の種類を分ける（README の表に基づく）。

パス目的
.trellis/spec/チーム標準・コーディングガイド。エージェントが自動で読み込める形にまとめる。
.trellis/tasks/PRD、タスク文脈、ステータス、レビュー記録、受入条件。
.trellis/workspace/開発者単位のジャーナル・判断・引き継ぎメモ。セッション間の連続性用。
.trellis/workflow.md計画・実装・検証・完了・学習の共有ライフサイクル。
プラットフォーム用生成物利用中のツール向けに、コマンド・hooks・skills・プロンプト・エージェント定義などを生成。
3 — 4 フェーズループ
README の How It Works

README は Plan → Implement → Verify → Finish の 4 フェーズを、スキルとサブエージェントの自動呼び出し付きで説明している。.trellis/workflow.md 側では Phase 1（Plan）・Phase 2（Execute）・Phase 3（Finish）に整理されている。

フェーズ主なスキル / エージェント成果物
Plantrellis-brainstorm、必要時 trellis-researchprd.md、implement.jsonl / check.jsonl に spec・調査を登録
Implementtrellis-implement サブエージェントPRD と curated context からコード（この段階では git commit しない）
Verifytrellis-check サブエージェントspec 照合、lint / type-check / test、可能な範囲で自己修正
Finish最終チェック後 trellis-update-spec学びを .trellis/spec/ へ還元し、次セッションの起点を更新
Plan → Implement → Verify → Finish
セッション終了: 作業完了またはコンテキスト上限に達したら /trellis:finish-work（プラットフォームにより /trellis-finish-work 等）でタスクをアーカイブし、ジャーナルを更新する（README How to Use）。

5 — インストール
前提と CLI

前提は Node.js 18 以上 と Python 3.9 以上（hooks と自動化スクリプト用）。グローバル導入と初期化の例は README に一致させている。

npm install -g @mindfoldhq/trellis@latest

# 開発者ワークスペースを作成
trellis init -u your-name

# 利用中のプラットフォームだけ初期化
trellis init --cursor --opencode --codex -u your-name

手順の詳細は Quick Start（install and first task）。

6 — 使い方
README How to Use
  1. 自然言語でやりたいことを述べる。
  2. trellis-brainstorm で 1 問ずつ要件を詰め、PRD が固まったら実装フェーズへ。
  3. trellis-implement が spec・lint・型・テストを自動チェックしながら実装。
  4. 完了またはコンテキスト上限で /trellis:finish-work — タスクをアーカイブしジャーナルを更新。

フック / 拡張があるプラットフォームではセッション開始時に文脈が自動ロードされる。手動入口が必要な環境では /start または /trellis:start でプロジェクト文脈を読み込む。

/start or /trellis:start   # プロジェクト文脈を読み込む
/trellis:finish-work        # 作業完了・セッション終了
7 — CLI（packages/cli）
主要コマンド

npm パッケージ @mindfoldhq/trellis のエントリは packages/cli。テンプレート取得・プラットフォーム別 configurators・マイグレーション manifest で .trellis/ と各ツール用ファイルを生成・更新する。

コマンド役割
trellis init初回セットアップ。spec / tasks / workspace / 選択プラットフォーム用生成物を配置。
trellis updateテンプレート差分を取り込み。workflow.md や hooks の更新を配布。
trellis uninstall管理対象パスの削除（manifest ベース）。

タスク操作は Python スクリプト（.trellis/scripts/task.py 等）が担い、task.py create / start / finish / archive でライフサイクルを管理する（workflow.md Task System）。

8 — CLAUDE.md との関係
FAQ の要約

CLAUDE.md、AGENTS.md、.cursorrules は有効な入口だが肥大化しやすい。Trellis はその外側に スコープ付き spec・タスク PRD・ワークフローゲート・ワークスペース記憶・プラットフォーム別生成物 を足す、という README FAQ の整理。

Claude Code 専用ではなく、チームとソロの両方を想定したプロジェクト層である点も FAQ で明示されている。
