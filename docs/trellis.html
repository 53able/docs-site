Trellis

mindfold-ai / npm @mindfoldhq/trellis — progressive specs, tasks, workflows, session memory

1 — 概要
何か
@mindfoldhq/trellis 0.5 skill-first AGPL-3.0 Node 18+ / Python 3.9+

Trellis は、リポジトリに組み込む チーム向け AI コーディングハーネス。通常 CLAUDE.md や AGENTS.md に詰め込む巨大システムプロンプトを、必要なときだけ読み込む 進行型の spec・タスク・ワークフロー・ジャーナル に分割する。

公式 README の位置づけどおり、Claude Code・Cursor・Codex・OpenCode・Pi Agent など複数のコーディングエージェントが、同じプロジェクトの正（チーム標準・タスク判断・手順・セッション記憶）を共有できる。

2 — .trellis/ のレイヤー
ディレクトリの役割

インストール後、リポジトリ直下に .trellis/ が置かれ、各サブツリーがエージェントが参照する知識の種類を分ける（README の表に基づく）。

パス目的
.trellis/spec/チーム標準・コーディングガイド。エージェントが自動で読み込める形にまとめる。
.trellis/tasks/PRD、タスク文脈、ステータス、レビュー記録、受入条件。
.trellis/workspace/開発者単位のジャーナル・判断・引き継ぎメモ。セッション間の連続性用。
.trellis/workflow.md計画・実装・検証・完了・学習の共有ライフサイクル。
プラットフォーム用生成物利用中のツール向けに、コマンド・hooks・skills・プロンプト・エージェント定義などを生成。
3 — コアループ
README で示される 6 手順

タスクを PRD に落とし、関連 spec を注入し、境界の中で実装し、引き渡し前にチェックを走らせ、再利用できる学びを spec に還元し、次のエージェントが読む記録を残す、という短い循環。

PRD 化 → spec 注入 → 実装 → チェック → spec へ還元 → セッション記録

詳細なプロダクトモデルは 公式ドキュメント と Real-World Scenarios が一次情報。

4 — 対応プラットフォーム
マルチエージェント

単一ベンダー専用ではなく、プロジェクト層として複数 IDE・エージェントにまたがる。README では Claude Code、Cursor、Codex、OpenCode、Pi Agent などに言及があり、Supported Platforms で差分を整理している。

5 — インストール
前提と CLI

前提は Node.js 18 以上 と Python 3.9 以上（hooks と自動化スクリプト用）。グローバル導入と初期化の例は README に一致させている。

npm install -g @mindfoldhq/trellis@beta

# 開発者ワークスペースを作成して開始
trellis init -u your-name

手順の詳細は Quick Start（install and first task）。

6 — 最初のタスク
0.5 の skill-first 動線

フックまたは拡張が使えるプラットフォームではセッション文脈が自動ロードされる。手動エントリが必要な環境では、README のとおり /start または /trellis:start からプロジェクト文脈を読み込む。

/start or /trellis:start   # プロジェクト文脈を読み込む

自然言語でタスクを述べると、ブレインストーミング・spec 読込・実装チェック・ナレッジ記録などのスキル経路にルーティングされる。README では continue でタスクを進め、人間テストとコミット後に finish-work を使う流れに触れている。

7 — CLAUDE.md との関係
FAQ の要約

CLAUDE.md、AGENTS.md、.cursorrules は有効な入口だが肥大化しやすい。Trellis はその外側に スコープ付き spec・タスク PRD・ワークフローゲート・ワークスペース記憶・プラットフォーム別生成物 を足す、という README FAQ の整理。

Claude Code 専用ではなく、チームとソロの両方を想定したプロジェクト層である点も FAQ で明示されている。
