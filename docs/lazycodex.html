LazyCodex

Codex 向けに oh-my-openagent（OmO） をパッケージ化したエージェントハーネス。npm パッケージ lazycodex-ai が npx 経由で OmO の Codex インストーラを呼び出す。

1 — 概要
プロジェクトの位置づけ
OmO コアエンジン
Codex 対象プラットフォーム
$ スキル・コマンド接頭辞
0 グローバル install 不要

LazyCodex は複雑なコードベース向けのエージェントハーネスを、Codex 上でそのまま使える形にまとめた配布レイヤ。プロジェクトメモリ、計画、実行、検証完了までをスキル・フック・マルチエージェントで扱う。LazyVim が lazy.nvim をまとめたのと同様に、OmO の品質バーを Codex ユーザー向けに一括導入する。

公開サイトは lazycodex.ai。詳細ドキュメントも同ドメイン配下。

2 — インストール
推奨: npx インストーラ

グローバル install は不要。npx で常に最新の OmO インストーラを呼ぶ。

npx lazycodex-ai install

内部では npx --yes --package oh-my-openagent omo install --platform=codex に相当する。TUI なし・自律モード:

npx lazycodex-ai install --no-tui --codex-autonomous

bin/lazycodex-ai.js は install 以外のサブコマンドも oh-my-openagent の omo CLI へ透過的に転送する。

インストール確認と削除

コマンド役割
npx lazycodex-ai doctorプラグインキャッシュ、フック、MCP、エージェント、設定の健全性レポート
npx lazycodex-ai uninstallプラグインキャッシュ・bin リンク・エージェントロール・~/.codex/config.toml の管理セクションを削除

実験的: Codex マーケットプレイス

主経路は npx のまま。Codex 内 /plugins または CLI からマーケットプレイス sisyphuslabs 経由でも omo を追加できる。

codex plugin marketplace add https://github.com/code-yeongyu/lazycodex
codex plugin add omo@sisyphuslabs
フック承認: マーケットプレイス経路では起動時レビューで omo フックを承認するまでフックは動かない。初回承認後はバックグラウンドで bootstrap が走り、完了後にセッション再起動が必要。自律モードは明示的に --codex-autonomous を選ぶ。
3 — コマンド
Codex コンポーザで呼ぶ $ コマンド
入力概要
$ulw-loop "task" [--completion-promise=TEXT] [--strategy=reset|continue] Oracle 検証付きの自己参照ループ。ultrawork モードは最大 500 反復、通常は 100 反復で打ち切り。
$ulw-plan "what to build" Prometheus 戦略プランナー。plans/<slug>.md に計画を書き、プロダクトコードは変更しない。
$start-work [plan-name] [--worktree <path>] 計画のチェックリストを Boulder 進捗で実行し、全項目完了で ORCHESTRATION COMPLETE を出力。

Composer で $ を入力するとインストール済みスキル一覧（init-deep、ulw-loop など）が開く。セッション中のフックは LazyCodex(<version>): ... 形式でステータスを表示する。

4 — ワークフロー
典型的な作業の流れ
$init-deep → $ulw-plan → $start-work → $ulw-loop / review-work
  1. プロジェクトメモリ: $init-deep が階層的 AGENTS.md を生成。複雑なディレクトリをスコアリングし、コード近傍にガイダンスを置く。
  2. 計画: 実装前に判断が要るとき $ulw-plan。出力は plans/ 配下の Markdown のみ。
  3. 実行: 計画が固まったら $start-work。チェックリストを durable な進捗で消化する。
  4. 検証完了: 証拠ベースで終わらせるとき $ulw-loop。実装後の多角的レビューは review-work スキル。
使い分け: 計画だけ → $ulw-plan。計画済みの一括実行 → $start-work。終了条件が曖昧な反復作業 → $ulw-loop。

サブエージェントロール

~/.codex/agents/ にロールが展開される。Codex の spawn_agent で agent_type を指定して子エージェントを起動する。

spawn_agent({
  "message": "TASK: map the auth flow end to end.",
  "agent_type": "explorer"
})

利用可能なロール例: explorer librarian plan momus metis codex-ultrawork-reviewer

agent_type パラメータが無い Codex ビルドでは、message 内でロールを記述するフォールバックにスキル側が対応する。

omo プラグイン構成（抜粋）

ローカル Codex プラグイン名は omo。内部コンポーネントは components/ 配下で分離されている。

  • ulw-loop / ultrawork — ループと ultrawork モード
  • lsp / lsp-daemon / lsp-tools-mcp — LSP 連携
  • rules / comment-checker — 指示と編集後チェック
  • bootstrap — 初回セットアップ（config、エージェントロール、sg バイナリなど）
  • telemetry — 匿名 DAU イベント（オプトアウト可能）

テレメトリはプロンプト・ソースファイル・パスを送らない。無効化例:

export OMO_CODEX_DISABLE_POSTHOG=1
export OMO_CODEX_SEND_ANONYMOUS_TELEMETRY=0

7 — モデルルーティング
タスク別モデル選択

OmO は全サブタスクに最上位モデルを使い続けない。タスクカテゴリとフォールバックチェーンでモデルを切り替え、クォータを抑えつつ品質を保つ。

カテゴリ向き例（README 記載）
quick小さな編集・軽量作業gpt-5.4-mini
ultrabrain難しい論理・深い推論高推論 GPT（xhigh 等）
agentic codingツール連鎖・実装gpt-5.3-codex など Codex 向けモデル

定義の参照元は OmO 側の openai-categories.ts と model-requirements.ts。セッション中に複数の GPT 系モデル名が出ても、ベンチマーク駆動の意図的ルーティングである。
