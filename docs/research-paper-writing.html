Research Paper Writing

research-paper-writing skill — ML / CV / NLP 論文を reviewer-friendly に仕上げるための構造化スキル

1 — 概要
このスキルとは

ML・CV・NLP 系の研究論文を査読者の視点で書き直し、first-impression・logical flow・evidence-backed claims を高めるための AI スキル。 彭思達（Peng Sida）教授の公開ノートを基に整理・構造化したもの。

主な使いどころ
  • Abstract / Introduction の下書き・改稿
  • Method セクションのモジュール設計記述
  • 段落フローとクレーム-証拠の整合チェック

視覚品質の原則
  • Teaser figure と pipeline figure を必ず用意
  • 表は minimal-ink で readable に
  • 書式と命名を論文全体で統一する
  • 視覚品質を「装飾」ではなく「コンテンツ」と扱う
出力の形式
  • コンパクトなセクションアウトライン（3〜7 箇条）
  • 段落の役割を明示した改稿テキスト
  • clarity・flow・unsupported claims のチェックリスト
  • Claim → Evidence → Status のマッピング表
2 — コアワークフロー
Step 01
論文ストーリーの明確化
文章レベルの修正より先に、論文全体の主張・構造・流れを整理する。
Step 02
セクション別ガイドの参照
references/ 内の該当セクションガイドだけをロード。全ガイドを一度に読まない。
Step 03
段落単位の改稿
1 メッセージ = 1 段落。1 往復に 1 段落だけを改稿する。
Step 04
逆アウトライニング
各セクション執筆後に reverse outlining を実行し、topic sentence と thesis のマッピングを確認。
Step 05
クレーム-証拠チェック
Abstract・Introduction の全主張が実験結果で裏付けられているか確認。
Step 06
Adversarial Paper Review
references/paper-review.md の 5 次元チェックリストで査読者視点の自己レビューを実施。
グローバル原則 9 箇条
  1. 11 段落 = 1 メッセージのみ。
  2. 2段落の最初の文でメッセージを明示する。
  3. 3名詞は自己完結させ、新語は使う前に定義する。
  4. 4文と文の接続に明示的な関係（原因・対比・結果・精緻化）を持たせる。
  5. 5懐疑的な査読者として adversarial self-review を繰り返す。
  6. 6視覚品質（図表）をコアコンテンツとして扱う。
  7. 7clean な teaser figure と pipeline figure を使う。
  8. 8表は readable かつ minimal-ink で作る。
  9. 9書式を論文全体で一貫させる。
3 — パラグラフ品質チェック

「この段落は流れているか？」と問われたとき、3 ステップで素早く判定する。

Step 1

外部読者として読む

  • 段落に 1 つの明示的なメッセージがあるか？
  • 最初の文が段落の目的を述べているか？
  • 全キーワードが文脈なしで読めるか？
  • 各文が前の文と明示的な関係（原因・対比・結果・精緻化・例示）で繋がっているか？
Step 2

逆アウトライニング

  • セクションの thesis / main claim を書き出す。
  • 各段落の topic sentence を書き出す。
  • 各段落の evidence / explanation を箇条書きにする。
  • topic sentence → thesis、evidence → topic sentence のマッピングを確認。マッピングできない段落は修正 or 削除。
Step 3

フロー修正

  • それでも流れが弱い場合は、仮の section header と明示的な transition phrase を追加。
  • 修正後、不要な header を削除して最終化する。
参照: references/does-my-writing-flow-source.md
4 — Abstract ガイド
書く前の必須設問: (1) 解決する技術的問題と、解が確立されていない理由は何か？ (2) 技術的貢献は何か？ (3) なぜ本質的に機能するのか？ (4) 技術的優位性と新知見は何か？
Version 1

Challenge → Contribution

  1. タスク
  2. 先行研究の技術的課題
  3. 課題を解決する技術貢献（1〜2 文）
  4. 貢献の利点
  5. 実験サマリー

技術的な用語名だけ言及し、詳細な手順は説明しない。

Version 2

Challenge → Insight → Contribution

  1. タスク
  2. 先行研究の技術的課題
  3. 課題を解決する insight（1 文）
  4. insight を実装する技術貢献（1〜2 文）
  5. 技術的新規性の利点
  6. 実験サマリー
Version 3

Multiple Contributions

  1. タスク
  2. 先行研究との対比（任意、1 文）
  3. 貢献1 + 技術的優位性
  4. 貢献2 + 技術的優位性
  5. 貢献3 + 技術的優位性
  6. 実験サマリー

「貢献 + 優位性を 1 文で表現する」能力が重要。

Abstract 品質チェック
  • 1 回読んでタスク・課題・貢献・実験結果が特定できるか？
  • 全主要クレームが実験で裏付けられているか？
  • 技術用語が自己完結しており、読みやすいか？
  • 1 文に複数のメッセージが混在していないか？
5 — Introduction ガイド

Introduction は「後ろから考え、前に向かって書く」。まず backward reasoning で核心を掴み、forward story として展開する。

Backward Reasoning（先に答える）

  1. 解決する技術的問題と、確立された解がない理由は？
  2. pipeline の貢献（新タスク・新手法・新指標・新技術）は？
  3. 貢献の利点と技術的課題を解決できる理由・新知見は？
  4. 先行手法をどう使って課題と新知見へ読者を誘導するか？

Forward Story（この順に書く）

  1. 論文のタスクを紹介する。
  2. 先行手法を使って解決した技術的課題へ導く。
  3. 技術的課題を解決する xx 個の貢献を提示する。
  4. 貢献の技術的優位性と新知見を明示する。
Introduction ロジックマップ

左側：思考の論理連鎖。右側：実際のセクション構成。両者のマッピングが Introduction の骨格を作る。

セクション骨格（LaTeX） 
\section{Introduction}
% Part 1: Task and application (target metrics)
% Part 2: Technical challenge for previous methods
%         (limitation + technical root cause)
% Part 3: Our pipeline solving the challenge
%         (technical contributions + why it works)
% Part 4: Additional contributions and impact
% Part 5: Experiments
6 — Method ガイド

Method は「pipeline figure スケッチ → サブセクション分割 → 各モジュールの 3 要素記述」の順で書く。

Method 執筆ステップ 
flowchart LR
    A["1. Pipeline figure を描く"] --> B["2. サブセクション構成を決める"]
    B --> C["3. 各サブセクションを計画\n（motivation / design / advantage）"]
    C --> D["4. Module design から書く\n（具体的な骨格）"]
    D --> E["5. Motivation と Technical Advantage を追加"]
要素 1

Module Design

  • 表現・ネットワーク・データ構造の詳細を記述。
  • forward process を明確に: input → step 1 → step 2 → output。
要素 2

Motivation

  • このモジュールが必要な理由を説明する。
  • 問題駆動のロジックで: 問題 X があるため、モジュール Y を設計。
要素 3

Technical Advantage

  • 代替案に対して技術的優位性がある理由を説明。
  • 可能なら測定可能な挙動に紐付ける。
書く前の必須設問: (1) pipeline 内のモジュールは何か？ (2) 各モジュールについて — ワークフロー・必要性・機能する理由 — の 3 問を答える。答えをマインドマップまたは表で整理してから執筆する。
7 — 論文レビュー（Adversarial Review）

提出前に査読者の立場で論文を読み、reject リスクを事前に潰す。主要クレームは全て技術的に正確かつ実験証拠で裏付けられていなければならない。

採択される論文の条件

十分な貢献

新タスク / 新 pipeline / 新モジュール / 新設計選択 / 新実験知見 / 新知見のいずれか。

公平な比較下での優位性

先行手法との公正な比較実験で経験的に優位であること。

十分な比較実験とアブレーション

主要設計選択の ablation study と重要 baseline の比較。

Reject 次元 典型的な失敗シグナル
1. 貢献が不十分 対象の失敗ケースが広く知られている / 提案技術はすでに well-explored で改善が予測可能。
2. 記述が不明瞭 技術的詳細が不足で再現不可能 / モジュールの motivation が不明確。
3. 実験効果が弱い 先行手法との改善が僅差 / 絶対性能が目標会議に届いていない。
4. 評価が不完全 ablation study 不足 / 重要 baseline や評価指標が欠落 / データセットが単純すぎる。
5. 手法設計に問題 実験設定が非現実的 / 技術的欠陥 / 頑健性が低い / 新制約が利点を上回る。

5 次元セルフレビュー

次元 1

Contribution

  • 読者に与える新知識は何か？
  • 非自明な技術的アイデアか？
  • 少なくとも 1 つの明確な novelty があるか？
次元 2

Writing Clarity

  • 手法は論文だけで再現可能か？
  • 全モジュールの motivation が明示されているか？
  • 各段落が 1 つの明確なメッセージを持つか？
次元 3

Experimental Strength

  • 強い baseline に対して意味のある改善があるか？
  • 複数データセット / 指標で一貫した改善か？
  • 強みと失敗ケースを誠実に報告しているか？
次元 4

Evaluation Completeness

  • 全主要設計選択の ablation があるか？
  • 公正な設定下で最新 baseline が含まれているか？
  • 評価指標は標準的で十分か？
次元 5

Method Soundness

  • 実験設定は実用的か？
  • 頑健性が高く、ハイパーパラメータ調整に依存しないか？
  • 追加制約より利点が大きいか？
Adversarial Workflow
  1. 懐疑的な査読者として論文を読む。
  2. 全設問に論文内の証拠で答える。
  3. pass / needs revision / needs experiment で分類。
  4. 未解決の major reject risk がなくなるまで改稿。
