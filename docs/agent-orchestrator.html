1 — 概要
何を解決するか

AI コーディングエージェントを 1 体ターミナルで動かすのは簡単だが、30 体を複数イシュー・ブランチ・PR にまたがって走らせると「誰がどのタスクを担当しているか」「CI が落ちたら誰に渡すか」「レビューコメントをどう返すか」という調整問題が生じる。agent-orchestrator（ao） はこの調整を自動化するオーケストレーション基盤。

各エージェントは独立した git worktree 上で動作し、CI 失敗時は自動でエージェントにログを送って修正させ、レビューコメントも自動ルーティングする。人間は ao start するだけで、あとは localhost:3000 の Web ダッシュボードで PR をレビュー・マージするだけで済む。

3 つの非依存性
非依存対応実装
エージェント非依存Claude Code / Codex / Aider / OpenCode
ランタイム非依存tmux / プロセス（Docker 拡張可）
トラッカー非依存GitHub / Linear / GitLab
2 — アーキテクチャ
コンポーネント構成

CLI → Core → Plugin スロット → Runtime（AI Agents on worktrees）という 3 層構造。外部サービス（GitHub / Linear）との接続は SCM・Tracker プラグインが担う。

3 — プラグインシステム
8 スロット / 20 実装

Core は 8 つのインターフェース（スロット）を定義し、各スロットの実装を差し替え可能にする Plugin-first アーキテクチャ。PluginModule インターフェースを実装して create() をエクスポートするだけで新規プラグインとして登録できる。

スロットインターフェース（メソッド）実装
Runtime create / destroy / sendMessage / getOutput / isAlive runtime-tmux、runtime-process
Agent getLaunchCommand / getActivityState / getSessionInfo agent-claude-code、agent-codex、agent-aider、agent-opencode
Workspace create / destroy / list workspace-worktree、workspace-clone
Tracker getIssue / generatePrompt / listIssues tracker-github、tracker-linear、tracker-gitlab
SCM detectPR / getPRState / getCIChecks / getReviews / mergePR scm-github、scm-gitlab
Notifier notify / notifyWithActions notifier-desktop、notifier-slack、notifier-webhook、notifier-composio
Terminal openSession / openAll terminal-iterm2、terminal-web
LifecycleManager start / stop / check ポーリングループで全セッションの状態遷移を管理する内部実装
Session ステータス遷移
spawning→ working→ pr_open→ ci_failed/ changes_requested/ approved→ mergeable→ merged

終端状態: killed / terminated / done / errored / cleanup

4 — Reaction エンジン
自動応答ループ

YAML で「CI 失敗 → エージェントに送る」「承認 + グリーン → 通知」などの自動反応をコードなしで設定できる。LifecycleManager がポーリングループで外部イベントを検出し、対応する Reaction を発火させる。

sequenceDiagram participant H as 人間 participant AO as ao CLI participant LM as LifecycleManager participant AG as AI Agent participant GH as GitHub H->>AO: ao start AO->>AG: Orchestrator Agent 起動 AG->>AO: ao spawn issue-101 AO->>AG: Worker Agent 起動 (新 worktree) AG->>GH: git push / PR 作成 GH-->>LM: CI 失敗 (poll) LM->>AG: CI ログを送信 AG->>GH: 修正 push GH-->>LM: レビューコメント LM->>AG: コメント転送 AG->>GH: 返答 push GH-->>LM: Approved + CI green LM->>H: デスクトップ通知 "マージ準備完了" H->>GH: マージ
Orchestrator / Worker の 2 層構造： Orchestrator エージェント自体も AI（ao CLI を操作するエージェント）。Worker エージェントを ao spawn するのも AI が担う。人間は Orchestrator を起動してあとはレビュー・マージするだけ。
5 — セットアップ
インストール
shell
npm install -g @composio/ao
# または
npx @composio/ao doctor --fix   # 依存関係の自動セットアップ

主な CLI コマンド（人間が使う）

shell
ao start                          # カレントリポジトリで起動
ao start https://github.com/org/repo  # リモートを clone して起動
ao stop                           # 全停止
ao status                         # セッション一覧表示
ao dashboard                      # ブラウザでダッシュボードを開く
ao doctor                         # 環境チェック（--fix で自動修正）

CLI コマンド（Orchestrator Agent が使う）

shell
ao spawn issue-101                # Worker Agent 起動
ao batch-spawn 101 102 103        # 複数 Agent を一括起動
ao send app-1 "修正してください"  # 実行中 Agent にメッセージ送信
ao session ls                     # セッション一覧
ao session kill app-1             # セッション終了
YAML 設定例（最小構成）
agent-orchestrator.yaml
port: 3000

defaults:
  runtime: tmux
  agent: claude-code
  workspace: worktree
  notifiers: [desktop]

projects:
  my-app:
    repo: owner/my-app
    path: ~/my-app
    defaultBranch: main
    sessionPrefix: app

reactions:
  ci-failed:
    auto: true
    action: send-to-agent
    retries: 2
  changes-requested:
    auto: true
    action: send-to-agent
    escalateAfter: 30m
  approved-and-green:
    auto: false   # true にすると自動マージ
    action: notify
データ保存先
  • 設定ファイル: agent-orchestrator.yaml（リポジトリ内）
  • セッションデータ: ~/.agent-orchestrator/{hash}-{projectId}/sessions/
  • git worktree: ~/.agent-orchestrator/{hash}-{projectId}/worktrees/

hash はコンフィグファイルのディレクトリパスから SHA-256 で生成（12 文字）。複数チェックアウト環境でも衝突しない。

前提条件： Node.js 20+ / tmux / gh CLI / 対象エージェント（Claude Code 等）のインストールが必要。ao doctor で自動確認・修正可能。
