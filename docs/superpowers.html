Superpowers

Composable skills for coding agents. v5.1.0 extends the same disciplined workflow to Claude Code, Cursor, Codex, Factory Droid, OpenCode, Gemini CLI, and GitHub Copilot CLI.

1 — 概要
何か

Superpowers は、コーディングエージェント向けのソフトウェア開発ワークフロー一式。組み合わせ可能な「スキル」と、エージェントがそれらを確実に使うための初期指示で構成され、コードを書く前に仕様を引き出し、設計を承認してからプランに落とし、TDD とレビューを通してマージ判断まで進める。

5.1.0最新確認版
8対応ハーネス
14コアスキル
0 depsbrainstorm server

エージェントは「何か作る」と検知すると、いきなりコードを書かずに「何をしたいか」を問いかける。仕様を固めて短いチャンクで確認し、承認後に実装プランを作る。「go」と言えば subagent-driven-development または executing-plans に進み、タスク単位で検証しながら作業する。

3 — 基本ワークフロー
7 ステップ

エージェントはタスクの前に必ず関連スキルを確認する。推奨ではなく必須のワークフローとして組み込まれる。

順スキル役割
1brainstormingコードを書く前に発動。曖昧なアイデアを問いで整理し、設計をセクションごとに示して承認を得る。仕様は inline self-review でプレースホルダ、矛盾、曖昧さを点検。
2using-git-worktrees設計承認後に発動。分離したワークスペースで新ブランチを作成し、セットアップとクリーンなテストベースを確認。
3writing-plans承認された設計に基づき、2〜5 分で完了するタスクに分解。各タスクにファイルパス・完全なコード・検証手順を記載し、プラン自体も自己点検。
4subagent-driven-development / executing-plans実行方式を選ぶ。推奨は fresh subagent per task でタスク間を連続実行（v5.1.0 で 3 タスクごとの強制停止を廃止）。代替は同一セッションでのバッチ実行＋人間チェックポイント。
5test-driven-development実装中に発動。RED-GREEN-REFACTOR を強制。失敗テストを書き、落ちることを確認し、最小コードで通す。テストより前に書いたコードは削除。
6requesting-code-reviewタスク間で発動。code-reviewer.md テンプレートで Task (general-purpose) をディスパッチし、プランとの整合性を深刻度別に報告。クリティカルな問題は進行をブロック。
7finishing-a-development-branchタスク完了時に発動。テスト確認後、マージ / PR / 維持 / 破棄の選択肢を提示し、ワークツリーを整理。

6 — 貢献時の注意
AI agent 向けの安全柵

Superpowers 本体への PR は厳しく審査される。直近 100 件の closed PR を監査したところ、却下率は約 94%。スキル本文は「文章」ではなく、エージェントの行動を変える実行可能なプロンプト資産として扱われる。

PR 前の必須チェック（v5.1.0）: PR テンプレート全項目を具体的内容で埋める。open/closed 両方で重複 PR を検索する。実在する問題か検証する。core に属する変更か判断する。モデル・ハーネス・プラグインを開示する。完全 diff を人間パートナーに見せて明示承認を得る。ターゲットブランチは dev（main ではない）。
避ける変更理由
ドメイン特化・プロジェクト特化のスキルcore は汎用スキルの場所。特定領域向けは standalone plugin が適切。
評価なしのスキル文面リライトRed Flags などの語彙は挙動調整済み。改善には before/after eval が必要。
架空の問題や理論上の修正実際のセッション、エラー、ユーザー体験に基づかない PR は却下されやすい。
複数の無関係な修正を束ねる1 PR 1 problem。まとめた変更はレビュー不能になりやすい。
第三者依存の追加新ハーネス対応を除き zero-dependency 方針。外部ツール依存は別プラグインへ。
新ハーネス PR にセッション transcript なし「Let's make a react todo list」で brainstorming が自動発火する end-to-end 証拠が必要。
7 — 哲学
設計思想
  • Test-Driven Development — 常にテストを先に書く
  • Systematic over ad-hoc — 当て推量よりプロセス
  • Complexity reduction — シンプルさを最優先
  • Evidence over claims — 主張の前に検証してから「完了」とする
