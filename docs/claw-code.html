2 — 哲学
人間が方向を示し、claws が実行する
"The real thing worth studying is the system that produced them: a clawhip-based coordination loop where humans give direction and autonomous claws execute the work."

ボトルネックの変化

エージェントシステムがコードベースを数時間で再構築できる今、希少資源はタイピング速度ではなくなった。

  • アーキテクチャの明確さ
  • タスク分解
  • 判断力・審美眼
  • 何を作る価値があるかの確信

ヒューマンインターフェースは Discord

tmux・Vim・SSH ではなく Discord チャンネルが本当のヒューマンインターフェース。

人はスマートフォンから一文を送り、その場を離れられる。claws がタスクを分解し、ロールを割り当て、コードを書き、テストし、完成すれば push する。

6 — ロードマップ
"Clawable" な実行ランタイムへ

主要ユーザーはフック・プラグイン・セッション・チャンネルイベント経由で接続されたclaws。人間ファーストの端末の仮定を排除し、機械可読性を向上させる。

1

Reliable Worker Boot

ワーカーの ready-handshake ライフサイクル、trust プロンプト自動解決、構造化セッション制御 API

2

Event-Native Clawhip Integration

型付きレーンイベントスキーマ（lane.started 〜 lane.failed）、失敗タクソノミー、アクショナブルサマリー圧縮

3

Branch/Test Awareness & Auto-Recovery

ステールブランチ検出、一般的な失敗からの自動回復レシピ、グリーンレベルコントラクト

4

Claws-First Task Execution

構造化タスクパケット形式、自律コーディング用ポリシーエンジン、claw ネイティブダッシュボード / レーンボード

5

Plugin & MCP Lifecycle Maturity

ファーストクラスのプラグイン/MCP ライフサイクルコントラクト、degraded-startup 構造化レポート、エンドツーエンドパリティ

MVP 成功基準
  • claw がワーカーを起動し、準備完了を確実に知れる
  • claws がタスクを誤ってシェルに入力しなくなる
  • ステールブランチの失敗がデバッグ時間を無駄にする前に特定される
  • clawhip が機械的状態を報告する（tmux prose ではなく）
  • MCP/プラグインの起動失敗が分類・表面化される
  • コーディングレーンが人間の監視なしに一般的な起動・ブランチ問題から自己回復できる
