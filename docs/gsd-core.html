gsd-core

GSD Core — Git. Ship. Done.（メタプロンプト・コンテキストエンジニアリング・仕様駆動開発）

1 — 概要
プロジェクト
@opengsd/gsd-core 1.3.x-dev MIT Node 22+

GSD Core（GitHub: open-gsd/gsd-core）は、Claude Code・OpenCode・Gemini CLI・Codex・Copilot・Cursor・Windsurf など複数の AI コーディングエージェント向けの、メタプロンプト・コンテキストエンジニアリング・仕様駆動開発のフレームワーク。コンテキスト窓が埋まるほど品質が落ちる context rot を避けるため、調査・計画・実行の重い処理を fresh context のサブエージェントに任せ、メインセッションを薄く保つ。

README のタグラインは Git. Ship. Done.。マイルストーン単位で Discuss → Plan → Execute → Verify → Ship のループを繰り返し、成果物はプロジェクト直下の .planning/ に Markdown と JSON で残す。

/gsd-*スラッシュコマンド
.planningプロジェクト状態
200kエグゼキュータ文脈
gsd-toolsCLI 補助
gsd-pi との関係: gsd-pi は open-gsd 組織のローカルファースト CLI コーディングエージェント（@opengsd/gsd-pi）。GSD Core は複数ランタイムにインストールされるワークフロー・コマンド・フックのパッケージで、v1 系のメンテナンスラインとして README でも位置づけられている。
2 — フェーズループと設計原則
5 ステップ（各フェーズ） 
flowchart LR
  D[Discuss] --> P[Plan]
  P --> E[Execute]
  E --> V[Verify]
  V --> S[Ship]
  S --> D2[次フェーズへ]
  1. 1Discuss — 実装方針（ライブラリ・エラー処理・境界）を計画前に CONTEXT.md へ記録。
  2. 2Plan — 調査・分解・plan-checker。並列 wave 向けに PLAN.md を依存順に並べる。
  3. 3Execute — 各エグゼキュータは fresh 200k 文脈で計画どおりにコミット。
  4. 4Verify — 要件・決定・成果の突合。差分があれば修正プラン。
  5. 5Ship — PR 作成とフェーズアーカイブ、STATE.md 更新。

視覚 UI が絡むフェーズでは、Discuss と Plan の間に任意の /gsd-ui-phase で UI-SPEC.md を挟める（the-phase-loop ドキュメント）。

Fresh context per agent

オーケストレータが起動する各エージェントはクリーンな文脈。累積会話による劣化を防ぐ。

Thin orchestrators

gsd-core/workflows/*.md は重い処理をせず、init・spawn・状態更新に徹する。

File-based state

DB なし。状態は .planning/ の人間可読ファイル。git 共有も可能。

Absent = enabled

config.json でキー欠落は有効扱い。無効化は明示的に指定。

Defense in depth

plan-checker、タスク単位コミット、実行後 verify、UAT まで多層。

マルチランタイム

インストーラが Claude / Codex / Gemini 等へコマンド・スキル・フックを配布。

3 — インストールと初回プロジェクト
推奨インストール 
npx @opengsd/gsd-core@latest

対話でランタイム（Claude Code、OpenCode、Gemini CLI、Kilo、Codex、Copilot、Cursor、Windsurf など）とグローバル／ローカルインストールを選ぶ。agents/ や commands/ を手でコピーしないこと — クロスランタイム互換のためインストーラ必須（README）。

初回プロジェクト:

/gsd-new-project
/gsd-new-project --auto @prd.md

生成物の例: PROJECT.md、REQUIREMENTS.md、ROADMAP.md、STATE.md、config.json、research/、CLAUDE.md（COMMANDS リファレンス）。

4 — アーキテクチャ
レイヤー（ARCHITECTURE.md 要約） 
flowchart TB
  U[User /gsd-command] --> CMD[commands/gsd/*.md]
  CMD --> WF[gsd-core/workflows/*.md]
  WF --> A1[Agent fresh context]
  WF --> A2[Agent fresh context]
  WF --> A3[Agent fresh context]
  A1 --> CLI[gsd-tools.cjs]
  A2 --> CLI
  A3 --> CLI
  CLI --> FS[".planning/ PROJECT ROADMAP STATE phases"]
  • Command layer — ユーザー向けエントリ。YAML frontmatter + プロンプトでワークフローを起動。
  • Workflow layer — 参照読み込み、エージェント spawn、状態遷移。
  • CLI tools — gsd-tools の command families、routing hub、observability seams。
  • File system — .planning/ に永続メモリ（セッション跨ぎ・/clear 後も残る）。

6 — コマンド（抜粋）

ランタイムごとに表記が異なる: Claude / Copilot 等は /gsd-command-name、Gemini は /gsd:command-name、Codex は $gsd-command-name（COMMANDS.md）。

名前空間ルータ（v1.40+）ルーティング先
/gsd-workflowdiscuss / plan / execute / verify / phase / progress
/gsd-projectマイルストーン、監査、サマリー
/gsd-qualitycode review、debug、audit、security、eval、ui
/gsd-contextmap、graphify、docs、learnings
/gsd-manageconfig、workspace、workstreams、ship、inbox 等
/gsd-ideateexplore、sketch、spike、spec、capture
コアコマンド用途
/gsd-new-project新規プロジェクト初期化（--auto @file で PRD 抽出）
/gsd-workspace隔離ワークスペースの作成・一覧・削除

公式ドキュメント索引: open-gsd/gsd-core/docs（チュートリアル・How-to・Explanation）。
