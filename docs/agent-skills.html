1 — 概要
何か

Agent Skills は、AI コーディングエージェントがシニアエンジニアと同じ品質規律でソフトウェアを開発できるように設計されたワークフロー集。各スキルは「手順（Process）・アンチ合理化表（Rationalizations）・検証要件（Verification）」の 3 要素で構成され、エージェントが最短ルートを選んで仕様・テスト・セキュリティレビューをスキップする問題を防ぐ。

19コアスキル
6開発フェーズ
7スラッシュコマンド
3エージェントペルソナ
設計哲学: スキルは「参照ドキュメント」ではなく「エージェントが従うワークフロー」。手順・チェックポイント・終了条件を持ち、SKILL.md がエントリポイント。サポート参照はトークン節約のために必要時だけロードする Progressive Disclosure を採用。
スキルの解剖学

すべてのスキルは一貫した構造を持つ。

┌─────────────────────────────────────────────┐
│  SKILL.md                                   │
│                                             │
│  ┌─ Frontmatter ─────────────────────────┐  │
│  │ name: lowercase-hyphen-name           │  │
│  │ description: Use when [trigger]       │  │
│  └───────────────────────────────────────┘  │
│                                             │
│  Overview         → 何をするスキルか           │
│  When to Use      → 発動条件                  │
│  Process          → ステップバイステップの手順   │
│  Rationalizations → スキップ言い訳 + 反論      │
│  Red Flags        → 問題のサイン               │
│  Verification     → 証拠要件（証明が必要）      │
└─────────────────────────────────────────────┘
2 — 開発ライフサイクル
6 フェーズ
DEFINE → PLAN → BUILD → VERIFY → REVIEW → SHIP

フルフィーチャーの標準的なスキル実行順:

 1. idea-refine                 → アイデアを精錬
 2. spec-driven-development     → 何を作るか定義
 3. planning-and-task-breakdown → 検証可能な粒度に分解
 4. context-engineering         → 正しいコンテキストをロード
 5. incremental-implementation  → スライスごとに実装
 6. test-driven-development     → 各スライスを証明
 7. code-review-and-quality     → マージ前レビュー
 8. git-workflow-and-versioning → クリーンなコミット履歴
 9. documentation-and-adrs      → 意思決定を記録
10. shipping-and-launch         → 安全にデプロイ
柔軟な適用: すべてのタスクがすべてのスキルを必要とするわけではない。バグ修正なら debugging-and-error-recovery → test-driven-development → code-review-and-quality の 3 スキルで十分なことが多い。

spec-driven-development コードの前に目的・コマンド・構造・テスト・境界を網羅した PRD を書く 新プロジェクト・機能・重大な変更を始めるとき

スキル何をするかいつ使うか
incremental-implementation 薄い垂直スライスで実装・テスト・検証・コミット。フィーチャーフラグ・安全なデフォルト 複数ファイルにまたがる変更

4 — 7 スラッシュコマンド
開発ライフサイクルにマップされたコマンド

スラッシュコマンドはエントリポイント。内部で適切なスキルを自動的にアクティベートする。

/spec
Spec before code
何を作るかを定義する。spec-driven-development スキルを起動。
/plan
Small, atomic tasks
実装方法を計画する。planning-and-task-breakdown スキルを起動。
/build
One slice at a time
インクリメンタルにコードを書く。実装系スキルを自動選択。
/test
Tests are proof
動作を証明する。test-driven-development スキルを起動。
/review
Improve code health
マージ前にレビュー。code-review-and-quality スキルを起動。
/code-simplify
Clarity over cleverness
コードを簡潔にする。code-simplification スキルを起動。
/ship
Faster is safer
プロダクションへデプロイ。shipping-and-launch スキルを起動。
自動スキル選択

コマンドを使わなくても、何をしているかに基づいてスキルが自動的にアクティベートされる。API 設計をしていれば api-and-interface-design が、UI を構築していれば frontend-ui-engineering が動く。

コアオペレーティング行動

すべてのスキルにまたがって常に適用される非交渉的な行動規範（using-agent-skills より）。

  • 仮定の表面化: 非自明な実装の前に仮定を明示し、承認を求めてから進む。
  • 混乱の積極的管理: 不整合や矛盾に直面したら STOP し、具体的な問いを立てて解消する。
  • 適切なプッシュバック: 問題のあるアプローチには直接指摘し、代替案を提案する。同意マシンにならない。
  • シンプルさの強制: 実装前に「これはより少ない行数でできるか？」を確認する。
  • スコープ規律: 指示されたものだけに触れる。近隣コードの無許可リファクタは行わない。
  • 検証の非省略: すべてのスキルには検証ステップがある。「それっぽい」は証拠にならない。
6 — セットアップ
対応ツール

Agent Skills は Markdown のスキルファイルで構成されており、システムプロンプトや指示ファイルを受け付けるあらゆるエージェントで動作する。

Claude Code Cursor Gemini CLI Windsurf GitHub Copilot OpenAI Codex その他エージェント
Claude Code（推奨）

Marketplace インストール

/plugin marketplace add addyosmani/agent-skills
/plugin install agent-skills@addy-agent-skills

ローカル / 開発

git clone https://github.com/addyosmani/agent-skills.git
claude --plugin-dir /path/to/agent-skills
Cursor

任意の SKILL.md を .cursor/rules/ にコピーするか、skills/ ディレクトリ全体を参照する。

# 特定スキルだけ使う場合
cp skills/spec-driven-development/SKILL.md .cursor/rules/spec.md

# ディレクトリ参照（推奨）
# .cursor/rules/ に skills/ を追加して全スキルを有効化
Gemini CLI / Windsurf / Copilot
ツールインストール方法
Gemini CLI gemini skills install https://github.com/addyosmani/agent-skills.git
またはネイティブスキルとして自動検出
Windsurf スキル内容を Windsurf のルール設定に追加。docs/windsurf-setup.md 参照。
GitHub Copilot agents/ の定義を Copilot ペルソナとして、スキル内容を .github/copilot-instructions.md に追加。
その他エージェント スキルはプレーン Markdown。システムプロンプトや指示ファイルとして渡す。docs/getting-started.md 参照。
Session Hooks

hooks/session-start.sh によるセッション開始フック。セッション開始時に自動的にスキルのコンテキストを設定し、エージェントがどのスキルを適用すべきかを認識できるようにする。

Contributing: スキルは 具体的（実行可能な手順・曖昧なアドバイスではない）・検証可能（明確な終了基準）・実戦検証済み・最小限であることが求められる。docs/skill-anatomy.md と CONTRIBUTING.md を参照。
