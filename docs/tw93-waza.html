Waza

tw93/Waza — Engineering habits you already know, turned into skills Claude can run.

1 — 概要
何か

Waza（技）は、エンジニアがすでに持っている習慣を 8 つの Skill に束ねたオープンソースのコレクション。モデルの出力を「たくさん書かせる」のではなく、目的と制約をはっきりさせて精度を上げることを狙う。

作者が位置づける三部作のひとつとして、Kaku（書く・コード）、Waza（技・習慣）、Kami（紙・ドキュメント）が並ぶ関係になっている。

ひとことで Claude Code Codex Desktop ZIP MIT

各 Skill はフォルダ単位で SKILL.md、参照資料、補助スクリプトを持つ。/health は Claude Code 専用で初回はサマリ監査が既定。ほかはホスト環境の質問・検索・フェッチに依存しないよう書かれている。

2 — 8 スキル
コマンドいつ役割（要約）
/think新規設計・判断の前問題を詰め、設計を圧力テストし、別エージェントが実装できる決定完了プランを出す。
/designフロント UIスクリーンショット駆動の美的反復を含み、中途半端なデフォルトではなく方向性のある UI を出す。
/checkタスク後・マージ前差分レビュー、プロジェクトの公開コンテキストからの制約抽出、リリースや issue 対応などの承認済みフォローを証拠付きで検証。
/huntバグ・回帰修正の前に根本原因を確認する体系的デバッグ。
/write文章中英のプロズを自然な語り口に整え、硬い AI 調を削る。
/learn未知ドメイン収集・咀嚼・アウトライン・肉付け・推敲・自己レビューまでの 6 フェーズ研究。
/readURL / PDFプラットフォーム別ルーティングでクリーンな Markdown に取得（GitHub・PDF・WeChat・Feishu など）。
/health環境監査CLAUDE.md・rules・skills・hooks・MCP を予算意識して点検（Claude Code のみ）。
Pre-build

think と design が「着手前」の軸。UI 話題は design を優先。

Post-build

check がレビュー・リリース・トリアージを包含。コードが壊れている話ではなく「仕上げ」の入口。

Diagnostic

hunt は現象ベース、health はツールチェーンまわりの異常。

3 — ルーティング（RESOLVER）
歧義の潰し方

公式の skills/RESOLVER.md は「どのトリガがどの Skill に落ちるか」の集中表と、複数候補があるときの優先ルールをまとめている。

  • 具体性優先: UI の話なら think より design。
  • URL: まず read で Markdown 化し、要約や長文研究なら続けて learn など。
  • 「見て」系: コードが届いた／PR 段階なら check。エラー・挙動おかしいなら hunt。
  • ツール異常 vs コード異常: Hook や MCP が怪しいなら health。
出力マーカー: すべての Skill が先頭行にインラインで 🥷 を付ける共通規約があり、検証スクリプトもこれをチェックする。
4 — インストールと更新
Claude Code（スラッシュ） 
npx skills add tw93/Waza -a claude-code -g -y

個別の /think … /health がグローバルに入る。

Codex 
npx skills add tw93/Waza -a codex -g -y
その他
  • Marketplace: /plugin marketplace add tw93/Waza から個別プラグインを入れる流れ。
  • Desktop: GitHub Releases の waza.zip を Customize → Skills から ZIP 作成。
  • 更新: npx skills update -g -y、または marketplace / ZIP の差し替え。
5 — 連鎖ワークフロー
自動ではつながらない

Skill はタスク完了で止まり、次はユーザが明示的に選ぶ。README が挙げる代表的な鎖は次のとおり。

目的例（矢印は人手の合図）
機能設計からマージまでthink → 承認 → 実装 → check → マージ
修正の shiphunt → 修正 → check → リリース／push フォロー
調査と文章化read → learn → write
デバッグ検証hunt → 修正 → check
6 — 設計メモ（リポジトリ側）
Skill と Script

判断や文脈適応が要るものは Skill。同じ入力で常に同じ出力なら script や rule。verify-skills.sh や rules/*.md が決定性側。

プロジェクト文脈

check は対象リポジトリの README・マニフェスト・CI・ユーザー指示など公開情報からコマンドやリスクを抽出する。非公開の機密パスは Skill に焼かない方針。

7 — オプション（Statusline / English Coaching）
Statusline

Claude Code 向けにコンテキスト窓・5 時間枠・7 日枠のクォータをコンパクトに表示。色は閾値で緑／黄／赤など。

curl -sL https://raw.githubusercontent.com/tw93/Waza/main/scripts/setup-statusline.sh | bash
English Coaching

オプションのルールで、プロンプトに英語の誤りがあるときだけ短い添削を末尾に付ける（中国語のみのプロンプトは対象外）。

curl -sL https://raw.githubusercontent.com/tw93/Waza/main/scripts/setup-english-coaching.sh | bash -s -- claude-code
