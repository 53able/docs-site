mattpocock/skills

README の “real engineering, not vibe coding” を、実務での選び方に変換する運用地図。巨大な自動運転ではなく、ズレを減らす小さなループを必要な場面に差し込む。

1 — 概要
Operating model

mattpocock/skills は、Claude Code などのエージェントに読み込ませる小さなスキル集。リポジトリ全体のプロセスを奪うのではなく、計画確認、共有語彙、TDD、診断、トリアージ、設計改善という「事故りやすい瞬間」にだけ介入する。

Source: README.md, .claude-plugin/plugin.json, .agents/invocation.md

Design stance

GSD / BMAD / Spec-Kit のように全体ワークフローを所有するより、ユーザーが制御を持ったまま必要なスキルを組み合わせる設計。失敗時にプロセス自体を直しやすい。

読み方: これはスキル一覧ではなくルーティング表。今起きている失敗を見て、入れるループを選ぶ。
2 — 呼び出しモデル
User-invoked vs model-invoked

v1.0 以降、スキルは 誰が到達できるか で分類される。旧来の Commands / Skills 区分は廃止。

種別到達方法description の向き例
user-invoked人間が slash 名を入力したときのみ人間向けの一行要約（disable-model-invocation: true）/grill-me, /to-prd, /ask-matt
model-invoked人間またはエージェントの自動選択モデル向けトリガー文（"Use when…"）/tdd, /domain-modeling, /grilling

user-invoked は model-invoked を呼べるが、別の user-invoked は呼べない。依存は /skill 形式の prose invocation で表現し、フォルダ横断リンクは使わない。共有語彙は /codebase-design と /domain-modeling に集約され、improve-codebase-architecture や grill-with-docs が参照する。

3 — メインフロー（idea → ship）
ask-matt router

/ask-matt が定義する主経路。コードベースがあるなら /grill-with-docs で合意と CONTEXT.md / ADR を残し、設計疑問は /handoff → /prototype → /handoff で runnable に解消する。複数セッションの build なら /to-prd → /to-issues まで同一 context windowを保ち、issue ごとに fresh session で /implement を走らせる。

1
Grill
/grill-with-docs で合意と domain docs。コードベースなしなら /grill-me。
2
Plan
複数 session なら /to-prd → /to-issues。単発ならそのまま implement へ。
3
Implement
/implement が issue を TDD slice で組み立て。内部で /tdd を呼ぶ。
4
Review
/code-review で Standards + Spec の二軸レビュー後 commit。

状況入口備考

4 — 失敗モード別ルート
Misalignment

Agent didn't do what I want

作る前の合意が薄い。/grill-me または /grill-with-docs で、設計ツリーを質問で潰してから実装へ進む。

Main skills: grill-me, grill-with-docs

Vocabulary drift

Agent is too verbose

同じ概念を毎回20語で説明している。CONTEXT.md と ADR に共有語彙を置き、会話とコードの命名を短くする。

Main skills: grill-with-docs, setup-matt-pocock-skills

Broken output

Code doesn't work

エージェントが動くフィードバックループを持っていない。/tdd は vertical slice で RED/GREEN を作り、/diagnosing-bugs は再現信号から原因へ進む。

Main skills: tdd, diagnosing-bugs

Entropy

Ball of mud

生成速度が設計負債も加速している。/to-prd は触る module を確認してから PRD を作り、/to-issues で vertical slice に分ける。/prototype で捨て UI/CLI を試し、/improve-codebase-architecture で deep module 候補を探す。

Main skills: to-prd, to-issues, prototype, improve-codebase-architecture

5 — セットアップ契約
Hard vs soft dependency

/setup-matt-pocock-skills は、他の engineering skills が共有する repo 固有の契約を作る。ADR は依存を 2 種類に分ける。Issue tracker や label mapping がないと誤った外部操作になるスキルは hard。domain docs がなくても動くが出力が鈍るだけのスキルは soft。

依存タイプ対象スキル必要な契約ない場合
hardto-issues, to-prd, triageIssue tracker と triage label vocabularyIssue 作成先や label が間違う
softdiagnosing-bugs, tdd, improve-codebase-architecture, codebase-design, domain-modelingdomain glossary と ADR動くが、語彙と判断が浅くなる

6 — 品質ループ
Feedback first

/diagnosing-bugs の中心は「まず高速で決定的な pass/fail signal を作る」こと。/tdd は一括で全テストを書く horizontal slice を避け、1つの振る舞いを RED → GREEN で通す。interface 設計の語彙は /codebase-design に委譲する。

Signaltest / CLI / browser / harness
Reproduceuser symptom を確認
Hypothesize3-5 個、予測つき
Instrument仮説に対応する probe
Fixregression test → fix
Postmortem防げた設計を確認
Code review gate

/code-review は固定点（commit / branch / tag）からの diff を Standards（repo 標準 + Fowler smell baseline）と Spec（originating issue / PRD との一致）の二軸で並列 sub-agent レビューする。implement の締め、または PR 前の単独実行に使う。

TDD contrast
やらないやる理由
全テストを先に書く1 test → 1 implementation想像上の振る舞いではなく、学びに反応した test になる
private method を検証するpublic interface の振る舞いを検証する内部リファクタで壊れない
mock で形だけ通すreal code path を通す本当の統合バグを拾える
7 — Issue 化フロー
PRD to slices

/to-prd は会話とコード理解を PRD にまとめ、/to-issues は tracer bullet の vertical slice に分解する。slice は schema/API/UI/tests の層ではなく、完了したら単体で demoable な細い経路として切る。

A
Conversation
既に話した内容と codebase 状態を合成する。
B
PRD
Problem、Solution、User Stories、Testing Decisions を作る。
C
Vertical slices
AFK / HITL と依存関係を明示して薄く分ける。
D
Implement
issue ごと fresh session で /implement → /code-review → commit。

外部由来の issue だけ needs-triage から triage state machine に乗せる。/to-issues 出力は agent-ready なので triage しない。

Triage state machine
Role意味次の動き
needs-triagemaintainer 評価待ちbug/enhancement と実装可能性を判断
needs-inforeporter から情報待ち具体質問に回答が来たら再評価
ready-for-agentAFK agent が拾えるagent brief を投稿

wontfix対応しない理由を残して閉じる
8 — 設計レビュー
Deepening opportunities

/improve-codebase-architecture は、浅い module を深くする候補を探す。語彙は /codebase-design、ドメインモデルは /domain-modeling から借りる。目的は美しい抽象化ではなく、locality と leverage。bug、変更、知識が散らばる場所を見つけ、interface の後ろに実装の複雑さを隠せるかを見る。

語彙見るポイント判断質問
Moduleinterface と implementation を持つ単位削除すると複雑さは消えるか、caller に散るか
Interfacecaller が知る必要のある契約test surface として十分に安定しているか
Seam振る舞いを差し替える場所adapter が2つ以上あり、現実の seam になっているか
Depth小さな interface の背後にある大きな振る舞いcaller は少ない知識で大きな効果を得ているか
運用のコツ: 迷ったら /ask-matt。実装前は grill-with-docs、ship は implement → code-review、壊れたら diagnosing-bugs、調査は research、泥団子化したら improve-codebase-architecture。順番より、失敗モードへの適合が大事。
