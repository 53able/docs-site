Agent Skills 設計ベストプラクティス

repos/skills-best-practices — agentskills.io 準拠のスキル設計・検証の要点

この資料について

ワークスペース内 repos/skills-best-practices の README とサンプル skill/（name: skill-creator）に沿ったビジュアル解説。公式の網羅ドキュメントは Claude Docs — Agent skills best practices、回帰防止には skillgrade、ベンチマークの参照先として SkillsBench が README で言及されている。

1 — 概要
設計思想

スキルはエージェントの「常時ロードされる脳」ではなく、YAML で発見され、必要なときだけ本文とリソースが読まれる手順書の束として扱う。README では SKILL.md をナビゲーションと高レベル手順の中枢としつつ、詳細は references/・assets/ に逃がしてトークンを抑える。

2 — ディレクトリ構成
標準レイアウト 
skill-name/
├── SKILL.md              # 必須: メタデータ + 中核指示（<500 行）
├── scripts/              # 小さな CLI（Python / Bash 等）— ライブラリ置き場ではない
├── references/           # 補助コンテキスト（スキーマ、チートシート）— 1 階層のみ
└── assets/               # 出力用テンプレや静的ファイル
  • references/ … references/foo/bar.md のようなネストは避け、フラットに 1 段だけ。
  • scripts/ … 毎回 LLM にパースを書かせると壊れやすい処理向け。リポジトリの本流ライブラリはここに詰め込まない。
  • assets/ … JSON スキーマ、テンプレ、画像などエージェントがコピー・参照する成果物用。
スキル内に置かないもの（README 明示）
人間向けの README.md / CHANGELOG.md / INSTALLATION_GUIDE.md、エージェント単体で既に安定している冗長な指示、長寿命のライブラリ実装。
3 — メタデータ（フロントマター）
name / description

トリガー前にエージェントが見えるのは、実質 name と description だけ、という前提で書く。曖昧だとスキルは「見えない」。

項目ルール
name1〜64 文字、a-z・数字・単一ハイフンのみ（連続ハイフン不可）。親ディレクトリ名と完全一致（例: angular-testing/SKILL.md なら name: angular-testing）。
description最大 1,024 文字、三人称で能力を述べる。ネガティブトリガー（「使うな」条件）を含め誤起動を減らす。

description の良い例 / 悪い例

評価例
悪い「React スキル」— 広すぎてルーティング不能。
良いTailwind で React コンポーネントのスタイルと UI ロジックを更新する。ユーザーがスタイルや UI ロジックの更新を求めるときに使う。Vue / Svelte / 素の CSS のみの案件では使わない、まで具体化（README の英語例に準拠）。
4 — プログレッシブ開示
コンテキスト窓の管理
  • SKILL.md は瘦せる … 500 行未満を目安に、手順の骨格と「いつどのファイルを読むか」に徹する。
  • 明示的パス … 参照は常に相対パス・スラッシュ区切り（OS 依存の表記にしない）。
  • JiT 読込 … 例: 「エラーコード一覧は references/auth-flow.md を読め」と SKILL.md に書き、エージェントが必要時だけ開く。
flowchart TD A["YAML name / description"] -->|"ルーティング"| B["SKILL.md 本体"] B -->|"番号付き手順"| C{"追加コンテキストが要る?"} C -->|Yes| D["references/ または assets/"] C -->|No| E["ツール実行・回答"] D -->|"JiT で読む"| E B -.->|"肥大時は分割"| D
用語の一貫性

同じ概念に template / html / markup を混在させない。ドメイン固有の正しい用語（Angular なら template など）に寄せると、エージェントのパターンマッチが安定する（README の指針）。

5 — 手順の書き方
LLM 向け文体
  • 番号付きステップ … 時系列で一意に追える手順にする。分岐は README の例のように「Step 2: 〜なら A、そうでなければ Step 3 へ」と明示。
  • 三人称命令形 … 「テキストを抽出せよ」。I will / You should は避ける。
  • 具体テンプレート … JSON や設定の形は長文で説明せず、assets/ に置き「構造をコピーせよ」と指示。エージェントはテンプレへのパターンマッチが強い。
6 — 決定的スクリプト
scripts/ の役割

複雑なパースや毎回同じボイラープレートは、テスト済みの小さな CLI に任せる。成功・失敗は stdout / stderr で人間が読めるメッセージにし、エージェントが自己修正しやすくする。

scripts/validate-metadata.py が見ること

  • name の長さ（1〜64）と ^[a-z0-9]+(-[a-z0-9]+)*$ 形式。
  • description の長さ（1,024 文字以下）。
  • 説明文に I / me / my / we / our / you / your が単語として含まれると STYLE WARNING（三人称推奨のヒューリスティック）。
python3 scripts/validate-metadata.py --name "my-skill" --description "Creates ... Don't use for ..."
7 — 検証ワークフロー
README の 4 段階

LLM がスキルの利用者になるため、ドラフト後は別チャットで次の順に洗う（README の Validation Guide）。

flowchart LR V1["Discovery Validation"] --> V2["Logic Validation"] V2 --> V3["Edge Case Testing"] V3 --> V4["Architecture Refinement"] V1 -.->|"description のみ貼付"| X1["誤検知・ブロードさを洗う"] V2 -.->|"全文 + ツリー"| X2["ステップ実行をシミュレート"] V3 -.->|"QA 役"| X3["失敗系・前提の穴を質問"] V4 -.->|"プログレッシブ開示で再構成"| X4["SKILL.md を縮小"]
段階目的（要約）
Discovery ValidationYAML の description だけを新規チャットに渡し、確実に起動すべきプロンプトと似ているが起動してはいけないプロンプトを生成させ、記述の広さを批判的にレビューさせる（README は agentskills.io 前提のサンプルプロンプト付き）。
Logic ValidationSKILL.md 全文とディレクトリツリーを渡し、具体リクエストに対してステップごとの内部独白（何をしているか・どのファイルか・曖昧さによる推測箇所）を書かせる。
Edge Case TestingQA 役に切り替え、スクリプト失敗・非対応設定など攻撃的な質問を出させ、先に修正せず一覧させる。
Architecture Refinement上記の回答を踏まえ、SKILL.md を高レベル手順だけに残し、巨大ブロックを references/ / assets/ へ移す。必要なら Error Handling 節を追加する。
評価・回帰防止
README は変更の効果測定のために skillgrade と SkillsBench を参照先として挙げている。
8 — 最終チェック（checklist.md）
skill/references/checklist.md

サンプル内のチェックリストは、デプロイ前にすべて Pass にする前提の監査項目。カテゴリごとの要点だけ抜粋する。

カテゴリ要点
Metadata & Discoveryディレクトリ名一致、1,024 字以内、Use when / Don't use、I・you 系の回避。
File Structurescripts/・references/・assets/ は 1 階層のみ、人間向け README 等なし、パスは /。
LogicSKILL.md 500 行未満、番号付き決定的ステップ、大きな塊は references / assets へ。
Scriptstiny CLI、成功は stdout・失敗は説明的 stderr、ライブラリ化しない。
Error Handlingよくある失敗と検証スクリプト実行ステップを SKILL.md に書く。
9 — サンプル skill-creator
repos/skills-best-practices/skill/

サンプルは name: skill-creator。手順はリポジトリの skill/SKILL.md の Step 1〜5 に従う。

  1. Step 1 — メタデータ定義 → python3 scripts/validate-metadata.py --name "..." --description "..."。エラー時は stderr を見て修正し再実行。
  2. Step 2 — 検証済み name でルートディレクトリを作り、scripts/・references/・assets/ を用意（README 等は作らない）。
  3. Step 3 — テンプレを起点に SKILL.md を執筆。本文では assets/skill-template.md とあるが、実ファイル名は assets/SKILL.template.md（大文字・拡張子の違いに注意）。
  4. Step 4 — 壊れやすい処理は scripts/ の単目的 CLI に閉じる。
  5. Step 5 — 幻覚ギャップの確認、相対パスとスラッシュ、references/checklist.md と突合。
Error Handling（サンプル SKILL.md）
メタデータ検証失敗時は stderr の種類（例: STYLE ERROR）に応じて書き換える。500 行超なら最大の手順ブロックを references/ に移し、SKILL.md からはそのファイルを読む命令だけにする。
