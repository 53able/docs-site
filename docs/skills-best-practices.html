Agent Skills 設計ベストプラクティス

repos/skills-best-practices — エージェント向けスキルの構造・メタデータ・検証の要点

この資料について

ワークスペース内 repos/skills-best-practices の README とサンプル skill/ を要約したビジュアル解説。Claude 公式のエージェントスキル解説への入口として、ディレクトリ規約・YAML の書き方・コンテキストを瘦せる考え方・LLM を使った検証手順を一枚に整理する。

1 — 概要
ゴール

スキルはエージェントの「脳みそ」ではなく、必要なときだけ読み込む手順書とリソースの束として設計する。人間向けの長文 README やライブラリ本体をスキル内に詰め込まず、発見しやすいメタデータとフラットな参照構造でルーティングとトークン消費を最適化する。

2 — ディレクトリ構成
標準レイアウト 
skill-name/
├── SKILL.md              # 必須: メタデータ + 中核指示（<500 行）
├── scripts/              # 決定的な小さな CLI（Python / Bash 等）
├── references/           # 補助コンテキスト（スキーマ、チートシート）
└── assets/               # 出力用テンプレや静的ファイル
  • SKILL.md … ナビゲーションと主要フロー。詳細は references/ へ逃がす。
  • references/ … 1 階層だけ（references/foo/bar.md のような深さは避ける）。
  • scripts/ … 毎回 LLM にパースを書かせない壊れやすい処理向け。ライブラリコードの置き場ではない。
スキル内に置かないもの（README の指針）
人間向けの README.md / CHANGELOG.md、エージェントが既に安定してこなす冗長な指示、長寿命のライブラリ実装。
3 — メタデータ（フロントマター）
name / description

エージェントがスキルを読む前に見えるのは YAML の name と description だけ、という前提で書く。

項目ルール
name1〜64 文字、小文字・数字・ハイフンのみ（連続ハイフン不可）。親ディレクトリ名と完全一致（例: angular-testing/SKILL.md なら name: angular-testing）。
description最大 1,024 文字、三人称で能力を述べる。ネガティブトリガー（「使うな」条件）を含め、誤起動を減らす。

description の良い例 / 悪い例

評価例
悪い「React スキル」（広すぎてルーティング不能）
良いTailwind で React コンポーネントのスタイルと UI ロジックを更新する。Vue / Svelte / 素の CSS だけの案件では使わない、のように具体化。
4 — プログレッシブ開示
コンテキスト窓の管理
  • SKILL.md は瘦せる … 500 行未満を目安に、手順の骨格と「いつどのファイルを読むか」に徹する。
  • 明示的パス … 参照は常に相対パス・スラッシュ区切り（OS 依存の表記にしない）。
  • JiT 読込の指示文 … 例: 「エラーコード一覧は references/auth-flow.md を読め」と SKILL.md に書き、エージェントが必要時だけ開く。
用語の一貫性

同じ概念に複数の言い回し（template / html / markup など）を混ぜない。ドメイン固有の正しい用語（Angular なら template など）に寄せると、エージェントのパターンマッチが安定する。

5 — 手順の書き方
LLM 向け文体
  • 番号付きステップ … 時系列で一意に追える手順にする。分岐は「Step 2: 〜なら A、そうでなければ Step 3 へ」のように明示。
  • 三人称命令形 … 「テキストを抽出せよ」— 一人称・二人称の説明口調は避ける。
  • 具体テンプレート … JSON や設定の形は文章で長く説明せず、assets/ に置いて「構造をコピーせよ」と指示。
6 — 決定的スクリプト
scripts/ の役割

複雑なパースや毎回同じボイラープレート生成は、テスト済みスクリプトに任せる。失敗時は stdout / stderr で人間が読めるメッセージを返し、エージェントが自己修正しやすくする。

サンプル skill/scripts/validate-metadata.py は、name と description の検証用 CLI として README・SKILL.md の手順に登場する。

7 — 検証ワークフロー
README に載る 4 段階
段階目的
Discovery Validationdescription だけを別チャットに貼り、起動すべき／すべきでないプロンプトを生成させて誤検知を洗う。
Logic Validation全文 SKILL.md とツリー構造を渡し、タスク実行をステップごとにシミュレートさせ、曖昧さによる「幻覚ギャップ」を列挙させる。
Edge Case TestingQA 役に切り替え、スクリプト失敗・非対応設定など攻撃的な質問を出させる。
Architecture Refinementプログレッシブ開示に沿って SKILL.md を分割し、肥大ブロックを references/ / assets/ へ移す。
ベンチマークの参照
README では SkillsBench（arXiv）や skillgrade を、スキル品質の評価・回帰防止の参考として挙げている。
8 — リポジトリ内サンプル（skill-creator）
skill/SKILL.md の流れ

repos/skills-best-practices/skill/ には、手順どおりにスキルを生成する例として skill-creator が入っている。

  1. メタデータ定義 → python3 scripts/validate-metadata.py --name "..." --description "..."
  2. scripts/・references/・assets/ を用意（人間向け README は作らない）
  3. assets/SKILL.template.md を起点に SKILL.md を執筆（本リポジトリの skill/assets/ に同ファイルあり。サンプル SKILL.md 本文は skill-template.md 表記のままなので、パスは実ファイルに合わせる）
  4. 壊れやすい処理は scripts に閉じ込める
  5. 最終チェックで references/checklist.md と突き合わせ

エラーハンドリング節では、メタデータ検証失敗時の修正と、500 行超過時の抽出・移動方針が短くまとまっている。
