repos/skills-best-practices — ディレクトリ構成・メタデータ・LLM 検証の要点

この資料について

ワークスペース内 repos/skills-best-practices の README・skill/（skill-creator）を元にしたビジュアル解説。ディレクトリ規約・YAML メタデータの書き方・JiT 読込によるコンテキスト削減・LLM を使った 4 段階検証を一枚に整理する。

1 — 概要
スキル作成の 5 ステップ
01
メタデータ初期化

name と description を定義し、validate-metadata.py で検証する。

02
ディレクトリ構成

scripts/・references/・assets/ の 3 フォルダを作成する。人間向け README は作らない。

03
SKILL.md 草稿

assets/SKILL.template.md を起点に 500 行未満で核心ロジックのみ書く。

04
スクリプト整備

壊れやすい処理・繰り返しロジックを scripts/ に閉じ込める。

05
LLM 検証

Discovery → Logic → Edge Case → Architecture の 4 フェーズで LLM と協力して品質を担保する。

スキル ≠ ライブラリ
スキルは「エージェントが必要なときだけ読み込む手順書とリソースの束」。人間向けの長文 README や長寿命ライブラリ実装をスキル内に詰め込まない。トークン消費の最小化が第一原則。
2 — ディレクトリ構成
標準レイアウト 
skill-name/
├── SKILL.md              # 必須: メタデータ + 中核指示（<500 行）
├── scripts/              # 決定的な小さな CLI（Python / Bash 等）
├── references/           # 補助コンテキスト（スキーマ、チートシート）
└── assets/               # 出力用テンプレや静的ファイル
フォルダ用途注意点
SKILL.md ナビゲーションと主要フロー。スキルの「脳」として機能する。 500 行未満を厳守。詳細は references/ へ。
references/ API ドキュメント、チートシート、ドメインロジック。 1 階層のみ（references/foo/bar.md 禁止）。
scripts/ 毎回 LLM に書かせると壊れる、繰り返しのパース・ボイラープレート処理。 ライブラリコードの置き場ではない。単一目的 CLI のみ。
assets/ 出力テンプレート、JSON スキーマ、静的ファイル。 エージェントが「構造をコピーせよ」と指示される対象。
スキル内に置かないもの
README.md / CHANGELOG.md / INSTALLATION_GUIDE.md などの人間向けドキュメント、エージェントが既に安定してこなす冗長な指示、長寿命ライブラリ実装。
3 — メタデータ最適化
フロントマターの原則

エージェントがスキルを読む前に見えるのは YAML の name と description だけ。ここが不正確だと「スキルが見えない」状態になる。

項目制約ポイント
name 1〜64 文字、小文字・数字・ハイフン（連続ハイフン不可） 親ディレクトリ名と完全一致（例: angular-testing/SKILL.md → name: angular-testing）
description 最大 1,024 文字 三人称で能力を述べ、ネガティブトリガー（「〇〇では使うな」）を含める
description の良い例 / 悪い例
評価記述例問題点 / 理由
悪い 「React スキル。」 広すぎてルーティング不能。トリガー条件もネガティブ条件も不明。
良い 「Tailwind CSS を使って React コンポーネントのスタイルと UI ロジックを更新する。Vue・Svelte・素の CSS だけの案件では使わない。」 何をするか（Creates）・ポジティブ条件（Use when）・ネガティブ条件（Don't use for）が揃っている。
4 — プログレッシブ開示
コンテキスト窓を守る 3 原則
原則実践方法
SKILL.md を痩せさせる 500 行未満を目安に。手順の骨格と「いつどのファイルを読むか」に徹する。大きなスキーマ・設定テンプレートは references/ や assets/ へ移す。
JiT 読込を明示 SKILL.md に「エラーコード一覧は references/auth-flow.md を読め」と書き、エージェントが必要時だけ開くよう誘導する。指示しない限りエージェントはファイルを読まない。
明示的パス 参照は常に相対パス・スラッシュ区切り（OS 依存の表記にしない）。例: references/schema.md、scripts/build.sh。
用語の一貫性

同じ概念に複数の言い回し（template / html / markup）を混ぜない。ドメイン固有の正しい用語（Angular なら template）に寄せると、エージェントのパターンマッチが安定する。

5 — 手順の書き方
LLM 向け文体の 3 ルール
ルール説明例
番号付きステップ 時系列で一意に追える手順にする。分岐は明示的に記述する。 「Step 2: ソースマップが必要なら ng build --source-map を実行せよ。そうでなければ Step 3 へ。」
三人称命令形 「Extract the text」のように直接命令する。一人称・二人称の説明口調は使わない。 悪い: 「I will extract…」「You should extract…」
良い: 「Extract the text…」
具体テンプレート JSON や設定の形は文章で説明せず、assets/ に置いて「構造をコピーせよ」と指示する。 「assets/vite.config.template.ts を読み、その構造をコピーして設定ファイルを生成せよ。」
6 — 決定的スクリプト
scripts/ の役割

複雑なパースや毎回同じボイラープレート生成は、テスト済みスクリプトに任せる。スクリプトは stdout / stderr で人間が読めるメッセージを返し、エージェントが自己修正できるように設計する。

validate-metadata.py — メタデータ検証 CLI

scripts/validate-metadata.py --name "[name]" --description "[description]" で実行。エラーは stderr に分かりやすいメッセージを返す。

import re, sys, argparse

def validate_metadata(name, description):
    errors = []

    # 1. name の長さチェック
    if not (1 <= len(name) <= 64):
        errors.append(f"NAME ERROR: '{name}' は {len(name)} 文字。1-64 文字必須。")

    # 2. name の文字チェック（小文字・数字・単一ハイフン）
    if not re.match(r"^[a-z0-9]+(-[a-z0-9]+)*$", name):
        errors.append(
            f"NAME ERROR: '{name}' に不正な文字が含まれます。"
            "小文字・数字・単一ハイフンのみ使用可。"
        )

    # 3. description の長さチェック
    if len(description) > 1024:
        errors.append(
            f"DESCRIPTION ERROR: {len(description)} 文字。1,024 文字以内必須。"
        )

    # 4. 一人称・二人称チェック（ヒューリスティック）
    first_person = {"i", "me", "my", "we", "our", "you", "your"}
    found = first_person & set(re.findall(r'\b\w+\b', description.lower()))
    if found:
        errors.append(
            f"STYLE WARNING: 一人称・二人称表現が含まれます: {found}。"
            "三人称命令形で記述してください（例: 'Creates...'）。"
        )

    if errors:
        print("\n".join(errors), file=sys.stderr)
        sys.exit(1)
    else:
        print("SUCCESS: Metadata is valid and optimized for discovery.")
        sys.exit(0)
7 — 検証ワークフロー（4 段階）

スキルの最終的なユーザーは LLM。だから検証も LLM と協力して行う。初稿が完成したら以下の 4 フェーズを順に実行する。

Phase 1 Discovery Validation — 誤起動を洗い出す

フロントマターだけを別チャットに貼り、「これが起動すべきプロンプト / 起動すべきでないプロンプト」を LLM に生成させる。誤検知のパターンを事前に特定する。

I am building an Agent Skill. Agents will decide whether to load this skill
based entirely on the YAML metadata below.

  name: angular-vite-migrator
  description: Migrates Angular CLI projects from Webpack to Vite and esbuild.
    Use when the user wants to update builder configurations, replace webpack
    plugins with rollup equivalents, or speed up Angular compilation.

Based strictly on this description:
1. Generate 3 realistic user prompts that should trigger this skill.
2. Generate 3 user prompts that sound similar but should NOT trigger this skill
   (e.g., migrating a React app to Vite, or just updating Angular versions).
3. Critique the description: Is it too broad? Suggest an optimized rewrite.
Phase 2 Logic Validation — 幻覚ギャップを特定する

SKILL.md 全文とディレクトリツリーを LLM に渡し、タスク実行をステップごとにシミュレートさせる。エージェントが推測を強いられる曖昧な箇所（幻覚ギャップ）を列挙させる。

Here is the full draft of my SKILL.md and the directory tree of its
supporting files.

  ├── SKILL.md
  ├── scripts/esbuild-optimizer.mjs
  └── assets/vite.config.template.ts

[SKILL.md の内容をここに貼り付ける]

Act as an autonomous agent that has just triggered this skill.
Simulate your execution step-by-step for the request:
"Migrate my Angular v17 app to Vite."

For each step, write out your internal monologue:
1. What exactly are you doing?
2. Which specific file/script are you reading or running?
3. Flag any Execution Blockers: Point out the exact line where you are
   forced to guess or hallucinate because instructions are ambiguous.
Phase 3 Edge Case Testing — 壊しにかかる

LLM に QA テスターの役割を与え、未対応設定・スクリプト失敗・暗黙的な前提などの攻撃的な質問を出させる。修正はまだしない — 問題を列挙させることに集中する。

Now, switch roles. Act as a ruthless QA tester.
Your goal is to break this skill.

Ask me 3 to 5 highly specific, challenging questions about edge cases,
failure states, or missing fallbacks in the SKILL.md. Focus on:

  - What if scripts/esbuild-optimizer.mjs fails due to a legacy
    CommonJS dependency?
  - What if the user's angular.json contains heavily customized Webpack
    builders (@angular-builders/custom-webpack) that Vite doesn't support?
  - Are there implicit assumptions about the user's Node environment?

Do not fix these issues yet. Just ask me the numbered questions
and wait for me to answer them.
Phase 4 Architecture Refinement — プログレッシブ開示を強制する

Phase 3 の回答をもとに、LLM に SKILL.md を再構成させる。大きな設定ブロックを references/ や assets/ に移動させ、トークン消費を最小化する。

Based on my answers to your edge-case questions, rewrite the SKILL.md file,
strictly enforcing the Progressive Disclosure design pattern:

1. Keep the main SKILL.md strictly as a high-level set of steps using
   third-person imperative commands (e.g., "Execute the esbuild script",
   "Read the Vite config template").

2. If there are dense rules, large vite.config.ts templates, or complex
   angular.json schemas currently in the file, remove them.
   Tell me to create a new file in references/ or assets/, and replace
   the text in SKILL.md with a strict command to read that specific file
   only when needed.

3. Add a dedicated Error Handling section at the bottom incorporating
   my answers about Webpack fallbacks and CommonJS resolution.
8 — デプロイ前チェックリスト

references/checklist.md に基づく最終監査。全項目が Pass でなければデプロイしない。

1. メタデータ & ディスカバリー
チェック項目基準
Namingname が 1-64 文字、小文字・数字・単一ハイフンのみ
Directory Matchname が親ディレクトリ名と完全一致
Description Lengthdescription が 1,024 文字以内
Trigger Optimization「Use when...」と「Don't use for...」が両方含まれている
Third-Person Tone「I」「me」「my」「you」「your」が含まれていない
2. ファイル構成 & パス
チェック項目基準
Flat Hierarchyscripts/・references/・assets/ 以下はすべて 1 階層のみ（ネスト禁止）
Standard Folders上記 3 フォルダ以外の独自フォルダを作らない
No Human DocsREADME.md・CHANGELOG.md・INSTALLATION_GUIDE.md が存在しない
Forward SlashesSKILL.md 内のすべてのパスがスラッシュ（/）区切り
3. ロジック & 指示（SKILL.md）
チェック項目基準
Lean ContextSKILL.md が 500 行未満
Imperative Mood指示が命令形（「Extract」「Run」「Validate」）で書かれている
Deterministic Steps番号付きの時系列ステップで、分岐が明示されている
Progressive Disclosure大きなスキーマ・テンプレートは references/ または assets/ に移動済み
Specific Terminologyドメイン固有の用語を一貫して使用（例: 「component」、「template」）
4. スクリプト & 決定性
チェック項目基準
CLI Designscripts/ のスクリプトが引数を取る小さな CLI として設計されている
Feedback Loop成功時は stdout、失敗時は stderr に説明的なメッセージを返す
No Library Codeスクリプトは単一目的。複雑なロジックはリポジトリ標準 CLI に委ねる
5. エラーハンドリング
チェック項目基準
Edge CasesSKILL.md に「Error Handling」セクションがあり、主要な失敗状態を網羅している
Validation該当する場合、検証スクリプトを実行するステップが含まれている
9 — サンプル（skill-creator）
skill-creator の構成

repos/skills-best-practices/skill/ には、本手順どおりにスキルを生成する skill-creator が入っている。

skill/
├── SKILL.md                  # skill-creator の手順書
├── scripts/
│   └── validate-metadata.py  # name / description の検証 CLI
├── references/
│   └── checklist.md          # デプロイ前チェックリスト
└── assets/
    └── SKILL.template.md     # 新規スキルの雛形
SKILL.template.md — 新規スキルの雛形

assets/SKILL.template.md を起点に SKILL.md を書く。このテンプレートがプログレッシブ開示と三人称命令形のパターンを示す。

---
name: [skill-name]
description: [Action-oriented capability description in the third person.
  Max 1,024 characters. Include positive triggers.
  Do not use for [explicit negative triggers].]
---

# [Skill Title]

## Procedures

**Step 1: [Action Phase]**
1. [Third-person imperative instruction, e.g., "Extract the query parameters..."]
2. [Instruction referencing an asset,
   e.g., "Read `assets/template.json` to structure the final output."]

**Step 2: [Action Phase]**
1. [Decision tree/conditional logic,
   e.g., "If source maps are required, run `scripts/build.sh`.
   Otherwise, skip to Step 3."]
2. [Instruction requiring JiT loading,
   e.g., "Read `references/auth-flow.md` to map the specific error codes."]
3. Execute `python scripts/[script-name].py` to [perform deterministic action].

## Error Handling
* If `scripts/[script-name].py` fails due to [specific edge case],
  execute [recovery step].
* If [condition B occurs], read `references/[troubleshooting-file].md`.
