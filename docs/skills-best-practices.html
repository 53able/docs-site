Agent Skills 設計ベストプラクティス

agentskills.io 仕様準拠 — 構造・メタデータ・指示・検証

1 — 概要
このガイドは何か

エージェント用スキル（Agent Skill）をプロ品質で作成するための凝縮版ガイドラインです。構造・メタデータ・指示の書き方・バリデーション手順を網羅しており、agentskills.io 仕様に完全準拠したスキルを一発で作れます。詳細は Claude 公式ドキュメント も参照してください。

<500SKILL.md 上限行数
<1024description 上限文字
1段サブディレクトリ深度
4手順バリデーションステップ
核心原則: スキルは人間のためではなく、LLM エージェントのために書く。README や CHANGELOG は禁止。情報は Just-in-Time で必要な時だけ読み込む設計にする。
2 — ディレクトリ構成
標準レイアウト

すべてのスキルは以下の 4 要素で構成されます。サブディレクトリは 1 段のみ（ネスト禁止）。

skill-name/
├── SKILL.md              # 必須: メタデータ + 中核手順 (上限 500 行)
├── references/           # 補足コンテキスト: API doc, チートシート, スキーマ
│   └── checklist.md
├── scripts/              # 実行可能な小さい CLI (Python / Bash)
│   └── validate-metadata.py
└── assets/               # 出力テンプレート / JSON スキーマ / 静的ファイル
    └── SKILL.template.md
各ディレクトリの役割
ディレクトリ用途禁止事項
SKILL.mdナビゲーションと高レベル手順の「脳」500 行超、密なスキーマの直接埋め込み
references/API doc、チートシート、ドメインロジック2 段以上のネスト
scripts/決定的な繰り返しタスクの小さい CLI汎用ライブラリコードの配置
assets/出力テンプレート、JSON スキーマ、静的ファイルREADME.md, CHANGELOG.md などの人向けドキュメント
3 — フロントマターの最適化
発見可能性を決める唯一のフィールド

エージェントがスキルを読み込む前に見るのは name と description のみです。この 2 フィールドが弱ければスキルは存在しないも同然です。

name フィールド

  • 1〜64 文字、小文字・数字・単一ハイフンのみ
  • 連続ハイフン禁止、親ディレクトリ名と完全一致が必須
  • 例: angular-vite-migrator は angular-vite-migrator/SKILL.md に置く

description フィールド — トリガー最適化

1,024 文字以内。第三者視点で書き、肯定トリガー（Use when...）と否定トリガー（Don't use for...）の両方を含める。

例
BAD "React skills." — 曖昧すぎる
GOOD "Creates and builds React components using Tailwind CSS. Use when the user wants to update component styles or UI logic. Don't use it for Vue, Svelte, or vanilla CSS projects."
NG パターン: I, me, my, you, your などの一人称・二人称を description に使うと Discovery Validation で失敗する。
4 — プログレッシブ・ディスクロージャー
コンテキストウィンドウを守る

スキルファイルはトークンを消費します。詳細情報は必要な時だけ読み込む設計（Just-in-Time Loading）が基本です。

原則実践方法
SKILL.md を 500 行以内に保つ大きな手順・スキーマ・設定テンプレートは references/ または assets/ に移す
フラットなサブディレクトリ1 段のみ。references/db/v1/schema.md のようなネストは禁止
Just-in-Time Loading「Step 3 で references/api-spec.md を読め」と明示しない限り、エージェントはそのファイルを見ない
明示的なパス指定常に相対パス + フォワードスラッシュ（OS 非依存）
作ってはいけないもの
  • README.md、CHANGELOG.md、INSTALLATION_GUIDE.md — 人向けのドキュメント
  • エージェントがすでに信頼できる動作をしているタスクへの冗長な指示
  • scripts/ に汎用ライブラリコードを置くこと（単一目的の小さい CLI のみ）
5 — 指示の書き方
LLM 向けに書く

Step-by-Step ナンバリング

ワークフローは 厳格な時系列番号順 で定義します。条件分岐がある場合は明示的に。

Step 2: ソースマップが必要なら `ng build --source-map` を実行。
        不要なら Step 3 へスキップ。

第三者命令形 (Third-Person Imperative)

指示は直接コマンドとして書きます。

例
NG"I will extract the text..." / "You should extract..."
OK"Extract the text from the specified element."

具体的なテンプレートを提供する

JSON や設定ファイルの出力フォーマットは文章で説明するより、assets/ にテンプレートを置いて「そのまま真似せよ」と指示する方が確実です。

用語の一貫性

同じ概念に複数の呼び名を使わない。Angular では "html/markup/view" ではなく "template" に統一するなど、ドメイン固有の最も正確な用語を使う。

6 — スクリプト活用
決定的操作はスクリプトに任せる

複雑なパース処理やボイラープレートは、毎回 LLM に書かせるより scripts/ に試験済みスクリプトを用意して実行させる方が安定します。

設計原則理由・実践
壊れやすい/繰り返し操作を移譲正規表現パース・DB クエリ・ファイル変換などは tested Python/Bash/Node スクリプトで実装
stdout/stderr を明示的に設計成功は stdout に人間が読める結果、失敗は stderr に具体的なエラーメッセージ → エージェントが自己修正できる
単一目的の小さい CLI引数を取る tiny CLI として設計。汎用ライブラリコードは禁止
エッジケースを丁寧に処理「このスクリプトが CommonJS 依存で失敗した場合は…」という Error Handling セクションを SKILL.md に追記
実例: scripts/validate-metadata.py --name "[name]" --description "[description]" — メタデータ検証をスクリプトで自動化し、エラー時は stderr で原因を出力して自己修正ループを作る。
7 — バリデーション手順
LLM と協働してバリデーションする

スキルの使い手は LLM です。最も効果的なバリデーションも LLM に依頼することです。以下の 4 ステップを順に実施します。

Step 1
Discovery Validation
フロントマターの YAML だけを新しい LLM チャットに貼り付け、「このスキルをトリガーすべき 3 プロンプト」と「すべきでない 3 プロンプト」を生成させる。誤検知・未検知を防ぐ。
Step 2
Logic Validation
SKILL.md 全体と対象ディレクトリ構造を LLM に渡し、「自律エージェントとして Step-by-Step で実行をシミュレートせよ」と依頼。推測が必要な箇所（Hallucination Gap）を全て列挙させる。
Step 3
Edge Case Testing
LLM を QA テスター役に切り替え、「このスキルを壊せ」と依頼。未サポート設定・環境仮定・フォールバック不足を 3〜5 個の具体的な質問として出させ、修正方針を確認する。
Step 4
Architecture Refinement
Edge Case の回答を踏まえ、プログレッシブ・ディスクロージャーを強制する形で SKILL.md を再構成させる。密なスキーマは references/ に移動し、メインを 500 行以内に保つ。
8 — チェックリスト
メタデータ & ディスカバリー
  • ✓name フィールドが 1〜64 文字かつ親ディレクトリ名と完全一致している
  • ✓description が 1,024 文字以内で肯定・否定トリガーを含む
  • ✓description に一人称・二人称が含まれない
ファイル構造 & パス
  • ✓すべてのサブディレクトリが 1 段のフラット構造になっている
  • ✓README.md、CHANGELOG.md など人向けドキュメントが存在しない
  • ✓SKILL.md 内のパスがすべて相対パス + フォワードスラッシュ
ロジック & 指示
  • ✓SKILL.md が 500 行以内に収まっている
  • ✓手順が番号付き時系列でまとめられ、条件分岐が明示されている
  • ✓大きなスキーマ・テンプレートが references/ または assets/ に分離されている
  • ✓SKILL.md に Error Handling セクションがある
スクリプト & 決定性
  • ✓scripts/ 内の各スクリプトが引数を取る tiny CLI として設計されている
  • ✓成功時は stdout、失敗時は stderr に具体的なメッセージが出る
