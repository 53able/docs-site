Phase 1

ドメイン分析

要求からドメイン・作業種別を抽出。既存エージェント/スキルとの衝突確認とコードベース探索を行い、ユーザーの技術習熟度を検出してトーンを調整する。

Phase 2

チームアーキテクチャ設計

6 パターンから構造を選択。エージェント分離の判断基準は専門性・並列性・コンテキスト・再利用性の 4 軸。デフォルトはエージェントチーム。

Phase 3

エージェント定義生成

.claude/agents/{name}.md に全エージェントを定義。すべて model: opus を使用。チーム通信プロトコル（SendMessage 先・受信範囲）も明示。

Phase 4

スキル生成

各エージェントのスキルを .claude/skills/{name}/SKILL.md に生成。Progressive Disclosure の 3 段階で context を管理。description は「pushy」に書く。

Phase 5

統合・オーケストレーション

TeamCreate でチームを構成し、TaskCreate で依存関係を管理。中間産出物は _workspace/ へ保存。フェーズ間の引き渡しはファイルベースが基本。

Phase 6

検証・テスト

構造確認 → 実行テスト（with/without 比較）→ トリガー検証 → ドライランの 4 段階。問題を一般化して修正し、反復改善ループで品質を高める。

6 アーキテクチャパターン

PL
Pipeline
エージェントが順次処理を引き継ぐ直列フロー。前段の出力が後段の入力になる。
→ 順次依存タスク（分析 → 執筆 → 校正）
FO
Fan-out / Fan-in
オーケストレーターが複数エージェントへ並列配布し、結果をマージ。独立タスクの並列実行に最適。
→ 独立並列タスク（複数ソース調査・多言語翻訳）
EP
Expert Pool
状況に応じて最適な専門エージェントを選択呼び出し。全員を並列起動せず、必要な専門家だけを呼ぶ。
→ 状況依存ルーティング（言語別・ドメイン別処理）
PR
Producer-Reviewer
生成エージェントと品質検査エージェントがペアで動作。不合格なら生成側へ差し戻しループ。
→ 品質保証が必要な生成タスク（コード・コンテンツ）
SV
Supervisor
中央スーパーバイザーが状態を管理し、進捗に応じてタスクを動的に分配する中央集権型。
→ 動的状態管理（フルスタック開発・データパイプライン）
HD
Hierarchical Delegation
上位エージェントが下位に再帰的委任。複雑な大規模タスクを階層的に分解して実行する。
→ 大規模タスク分解（マーケティング・複合研究）
04

スキル設計システム

Progressive Disclosure（段階的情報公開）
段階 ロード時期 サイズ目標
Metadata 常時コンテキスト常駐 ~100 語
SKILL.md 本文 スキルトリガー時 < 500 行
references/ 必要な時だけ読む 無制限
ディレクトリ構造
skill-name/
├── SKILL.md          ← 必須 · 500 行以内
│   ├── YAML frontmatter
│   │   ├── name: skill-name
│   │   └── description: "pushy に書く"
│   └── Markdown 本文
└── Bundled Resources （選択）
    ├── scripts/      ← 決定論的・繰り返しコード
    ├── references/   ← JiT ロード参照文書
    └── assets/       ← テンプレート・画像
Description — pushy トリガー設計
Description はスキルの唯一のトリガーメカニズム。Claude はトリガーを保守的に判断するため、積極的（pushy）に記述する必要がある。
評価 記述例
NG PDF 文書を処理するスキル
OK PDF 読み込み・テキスト抽出・ページ結合・分割・OCR 等すべての PDF 操作を実行。.pdf を言及または PDF 出力を要求したら必ずこのスキルを使う
作成 5 原則
  • Why 「なぜか」の理由を伝える — LLM はエッジケースでも正しく判断できる
  • Lean SKILL.md は 500 行以内。超過分は references/ に移動する
  • General特定例向けの狭い規則より原理を説明してオーバーフィットを防ぐ
  • Bundle 繰り返しコードは scripts/ にバンドル済みで提供する
  • Imperative「〜する」「〜せよ」の命令形で記述する
05

実行モード

Default Mode

Agent Teams

TeamCreate + SendMessage + TaskCreate でエージェントが自律協調。発見共有・対立議論・欠落補完が品質を底上げする。

  • 2 名以上のエージェントが協働するすべてのケース
  • SendMessage による直接エージェント間通信
  • TaskCreate / TaskUpdate で依存タスク管理
  • フェーズ間はファイルベースで産出物を引き渡す
  • セッション内でチームを解散・再構成可能
Alternative Mode

Subagents

Agent ツールで直接サブエージェントを呼び出す。エージェント間通信が不要なシンプルな構造向け。

  • エージェントが 1 名のみの場合
  • 結果の受け渡しだけで通信不要な構造
  • run_in_background=true で並列実行可能
  • 1 回限りのタスクや簡易自動化
チームサイズガイドライン
タスク規模 推奨チーム人数 メンバーあたり作業数 注意
小規模（5〜10 タスク） 2〜3 名 3〜5 件 —
中規模（10〜20 タスク） 3〜5 名 4〜6 件 —
大規模（20+ タスク） 5〜7 名 4〜5 件 調整オーバーヘッドに注意

3 名の集中したチームは、5 名の散漫なチームより優れた結果を出す。
