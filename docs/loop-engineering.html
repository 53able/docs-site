Loop Engineering

Design the system that prompts your agents — cobusgreyling/loop-engineering

1 — 概要
何か

Loop Engineering は、個別のプロンプトを書く代わりに、エージェントを反復的に動かす制御システム（ループ）を設計する実践の参照集。Grok・Claude Code・Codex・Cursor などの AI コーディングエージェント向けに、7 本の本番パターン、クローン可能な starter、3 つの npm CLI を公開している。

2 — 概念と語彙
ハーネスとループ

単一セッションの環境（ツール・権限・ルール）がAgent Harness。スケジュール・状態・検証チェーンを足したものがLoop。

Harness = 単一セッションのセットアップ
Loop    = Harness + スケジュール + 状態 + 検証チェーン

Intent Debt

セッションごとにエージェントは冷たく始まる。Skills で慣習・ビルド手順を一度書き、毎回読ませる。

Comprehension Debt

ループが速く ship するほど、自分で書いていないコードとの理解の差が広がる。ループの出力を読む運用が必要。

Orchestration Tax

並列エージェントのレビュー帯域・マージ競合・コンテキスト切替のコスト。Worktrees が機械的衝突を減らす。

Maker / Checker

実装者と検証者を分離。実装者が自分の宿題を採点しない構造が、無人ループの前提。

段階的ロールアウト

L1 — Report

レポートのみ。人間が判断。Week 1 のデフォルト。

L2 — Assisted

小さな修正を worktree + verifier 付きで実行。人間ゲートあり。

L3 — Unattended

allowlist 内の無人実行。検証と denylist が前提。

複数ループ併走時の優先度: CI Sweeper → PR Babysitter → Dependency Sweeper → Post-Merge / Changelog（オフピーク）→ Daily Triage（レポート）。

npx @cobusgreyling/loop-init . --pattern daily-triage --tool grok

# 2. トークン見積もり
npx @cobusgreyling/loop-cost --pattern daily-triage --level L1

# 3. 準備度監査
npx @cobusgreyling/loop-audit . --suggest

# 4. Week 1 はレポートのみ（Grok 例）
/loop 1d Run loop-triage. Update STATE.md. No auto-fix in week one.

監査スコアは空 → L1 → L2 と上がる想定。loop-budget.md でトークン上限、loop-run-log.md で実行履歴を追跡する。

8 — 安全と運用
制約と kill switch
  • 検証は人間の責任: 無人ループは無人のミスを増幅する
  • Denylist: コアドキュメント・audit ロジック等は自動変更禁止
  • Auto-merge: trivial dependency patch + verifier のみ例外
  • Kill switch: loop-pause-all ラベル、または STATE.md のフラグ
  • MCP スコープ: L1 では read + comment まで。信頼後に拡張
注意: サブエージェントと長時間ループはトークンコストが急増する。同じループでも結果はチームの判断次第 — ループは知らない、人間が知っている。
