iii

Worker / Trigger / Function で分散ワークを束ねるランタイム（iii-hq/iii）

1 — 概要
何を解くか

iii は、サービスを増やすたびに積み上がる「サービス間の接続コスト」を下げるためのインフラだ。公式 README の整理では、マイクロサービスやワーカーを足すほど潜在的な接続点が増え、運用・学習・プロンプト設計の負荷が二次的に増える問題を、登録さえすれば他ワーカーと即座にやり取りできる単一の実行モデルへ寄せる。

Workerプロセス単位の登録
TriggerHTTP / cron / キュー等
Function安定 ID の処理単位
Any言語・ランタイム非依存

CLI でプロジェクトを初期化し、同じディレクトリでエンジンを起動する流れがドキュメント化されている。エージェント向けには skills/ 配下の Skill 群を Skillkit から取り込む手順も README に載っている。

2 — 問題設定と設計方針
統合コストの見立て

各サービスは内部実装・ライフサイクル・失敗モードがバラバラになりがちで、新規サービス N 本目は既存との組み合わせを増やす。iii はこの「足し算」をワーカー登録と宣言的トリガーに畳み込み、4 本でも 20 本でも同じ操作感でスケールさせることを狙う。

観測性: すべての能力を同じ3語に写像すると、トレース・キュー・状態・ログを横断して追いやすい、という README 上の主張になっている（実装詳細は公式ドキュメント側）。
3 — 3 プリミティブ
Worker → Trigger → Function

公式の言い切りは次の三つに集約される。

  • Worker: エンジンに登録するプロセス。TypeScript の API、Python のパイプライン、Rust のマイクロサービスなど、形を問わず「ワーカー化」できる。
  • Trigger: 関数を起動する条件。直接呼び出し、HTTP、cron、キュー購読、状態変化、ストリームなど。ワーカー側で宣言し、ルーティング・シリアライズ・配信は iii 側が扱う、という役割分担。
  • Function: 名前空間::識別子 形式の安定 ID を持つ処理単位（例: content::classify）。入力を受け、任意で出力を返す。
Worker 登録 → Trigger 宣言 → Function 実行 → 他 Worker とも同一規約
