Skill Creator

agentskills.io 準拠 — Agent Skill 作成の 5 ステップガイド

1 — 概要
skill-creator とは

skill-creator は agentskills.io 仕様 に沿ってプロ品質の Agent Skill を作成するためのスキルです。メタデータの最適化・ディレクトリ構成・指示の書き方・バリデーションまでを 5 ステップで一貫して行います。

解説ページ（Agent Skills 設計ベストプラクティス）がベストプラクティスの原則を扱うのに対し、skill-creator スキル自体はその原則を実行する手順（Procedure）を提供します。

5作成ステップ数
<500SKILL.md 上限行
1段サブディレクトリ深度
4バリデーション観点
トリガー条件: 新規スキルディレクトリの作成・手順指示の草案・メタデータのディスカバリ最適化を行うとき。Don't use for 一般ドキュメント・非エージェント向けライブラリコード・README ファイル。
2 — ディレクトリ構成
標準レイアウト

すべてのスキルは以下の 4 要素で構成します。サブディレクトリは 1 段のみ（ネスト禁止）。人間向けドキュメント（README.md 等）は作成しない。

skill-name/
├── SKILL.md              # 必須: メタデータ + 中核手順 (上限 500 行)
├── scripts/              # 決定論的タスク用の小さい CLI
│   └── validate-metadata.py
├── references/           # API doc, チートシート, ドメイン知識 (1 段のみ)
│   └── checklist.md
└── assets/               # 出力テンプレート / JSON スキーマ / 静的ファイル
    └── SKILL.template.md
フォルダ 用途 読み込みタイミング
SKILL.md 高レベルな手順とナビゲーション。「ブレイン」 スキルトリガー時に常に読み込まれる
scripts/ 反復・脆弱な操作を担う単一目的スクリプト SKILL.md が明示的に実行を指示したとき
references/ スキーマ、チートシート、ドメイン知識 Just-in-Time — 手順が明示したときのみ
assets/ 出力テンプレート、JSON スキーマ、静的ファイル Just-in-Time — 手順が明示したときのみ
3 — 5 ステップ手順
実行フロー

以下のステップを厳密な時系列順で実行する。各ステップは前ステップの完了を前提とする。

Step 1
メタデータの初期化と検証
  1. ユニークな name を定義（1–64 文字、小文字・数字・単一ハイフンのみ）
  2. description を草案（最大 1,024 文字、第三者視点、ネガティブトリガー含む）
  3. 検証スクリプトを実行:
    python3 scripts/validate-metadata.py --name "[name]" --description "[description]"
  4. エラーがあれば stderr を確認し、合格するまで修正・再実行
Step 2
ディレクトリ構造の作成
  1. 検証済みの name でルートディレクトリを作成
  2. scripts/・references/・assets/ を初期化
  3. README.md・INSTALLATION.md 等の人間向けドキュメントは作成しない
Step 3
コアロジックの草案（SKILL.md）
  1. assets/SKILL.template.md を出発点として使用
  2. すべての指示を第三者命令形で記述（例: "テキストを抽出する"）
  3. プログレッシブ・ディスクロージャを徹底:
    • メインロジックは 500 行以内に収める
    • 大規模なスキーマは references/ へ移動
    • 必要時のみ読むよう明示的に指示する
Step 4
スクリプトの特定とバンドル
  1. 「脆弱」なタスク（正規表現、複雑なパース、定型ボイラープレート）を識別
  2. scripts/ ディレクトリに単一目的スクリプトを配置
  3. スクリプトは stdout/stderr でエージェントが自己修正できるよう設計
Step 5
最終ロジック検証
  1. SKILL.md 内の「ハルシネーションギャップ」（エージェントが推測せざるを得ない箇所）をレビュー
  2. すべてのファイルパスが相対パスかつフォワードスラッシュ（/）を使用していることを確認
  3. references/checklist.md に対して最終出力をクロスリファレンス
4 — SKILL.md テンプレート
assets/SKILL.template.md

Step 3 の出発点となるテンプレート。第三者命令形・JiT ローディング・エラーハンドリングを含む最小構成。

---
name: [skill-name]
description: |
  [能力を第三者視点で記述。最大 1,024 文字。
   ポジティブトリガー "Use when..." とネガティブトリガー
   "Don't use for..." を両方含める。]
---

# [Skill Title]

## Procedures

**Step 1: [Action Phase]**
1. [第三者命令形の指示。例: "クエリパラメーターを抽出する..."]
2. [アセット参照の指示。例: "assets/template.json を読んで最終出力を構造化する。"]

**Step 2: [Action Phase]**
1. [条件分岐。例: "ソースマップが必要な場合は scripts/build.sh を実行。不要なら Step 3 へ。"]
2. [JiT ローディング。例: "references/auth-flow.md を読んで対象エラーコードをマップする。"]
3. Execute `python scripts/[script-name].py` to [決定論的な処理を実行].

## Error Handling
* If `scripts/[script-name].py` fails due to [specific edge case], execute [recovery step].
* If [condition B occurs], read `references/[troubleshooting-file].md`.
5 — バリデーション
validate-metadata.py の使い方

メタデータの検証スクリプト。STYLE ERROR / LENGTH ERROR などを stderr に出力し、問題がなければ成功終了する。

# 基本的な使い方
python3 scripts/validate-metadata.py \
  --name "angular-vite-migrator" \
  --description "Migrates Angular CLI projects from Webpack to Vite..."

# エラー例: 一人称が含まれる場合
# STYLE ERROR: Description must be written in the third person.
# Found first-person pronouns: ["I", "my"]
合格するまで再実行: stderr の指示に従って name / description を修正し、Step 1 に戻って再実行する。合格後に Step 2 へ進む。
references/checklist.md — 最終監査

Step 5 で references/checklist.md を読んで全項目を確認する。5 観点 / 計 17 チェック項目。

  • 1
    メタデータ & ディスカバリ — name の命名規則、ディレクトリ名との一致、description 1,024 文字以内、"Use when" / "Don't use for" 両方含む、第三者視点
  • 2
    ファイル構造 & パス — サブディレクトリは 1 段のみ、標準フォルダのみ使用、README.md 等なし、すべてフォワードスラッシュ
  • 3
    ロジック & 指示（SKILL.md） — 500 行以内、命令形、番号付き手順、プログレッシブ・ディスクロージャ、ドメイン固有用語の統一
  • 4
    スクリプト & 決定論 — CLI 設計、成功/失敗のフィードバックループ、ライブラリコードを含まない
  • 5
    エラーハンドリング — "Error Handling" セクションあり、該当箇所でバリデーションスクリプト実行
6 — エラー処理
代表的な失敗パターン
パターン 原因 対処
Metadata Failure validate-metadata.py が STYLE ERROR を返す 一人称代名詞（I / me / my / you / your）を削除して第三者表現に書き直し、再実行
Context Bloat SKILL.md が 500 行を超える 最大の手順ブロックを references/ の新規ファイルに移動し、SKILL.md からは読み込み指示のみ残す
Hallucination Gap 手順が曖昧でエージェントが推測するポイントが生まれる Step 5 のロジック検証を再実施し、明示的な条件分岐・ファイル参照に修正
Deep Nesting references/ 以下でさらにサブディレクトリを作成している フラット構成（1 段のみ）に戻す。ネスト構造はエージェントに見えない
スクリプトに依存しすぎない: スクリプトはエージェントが生成するのではなく、事前に配置されたものを実行させる。長命なライブラリコードは scripts/ に置かず、リポジトリの標準 CLI に委ねる。
