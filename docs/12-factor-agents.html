12-Factor Agents

本番向け LLM アプリケーションの設計原則 — humanlayer/12-factor-agents

1 — 概要
CC BY-SA 4.0（本文） Apache 2.0（コード） 12factor.net 系譜

12-Factor Agents は、Dex Horthy（HumanLayer）が公開しているガイドです。12 Factor Apps の精神を踏まえ、本番の顧客に届けられる LLM ソフトウェアを作るための原則を 12 項目に整理しています。

主張の中心は次のとおりです。多くの「AI エージェント」製品は、プロンプトとツールの袋をループするだけの薄いラッパーではなく、大部分は決定的なソフトウェアであり、LLM 呼び出しは必要な箇所にだけ差し込む、という設計が実際に強い、というものです。

2 — 背景（ソフトウェアからエージェントへ）

ガイドは、ソフトウェアを有向グラフ（DAG）として捉える流れから始まります。従来のプログラムはフローチャートのように分岐と順序が決まっており、Airflow や Prefect などの DAG オーケストレータは観測性・リトライ・モジュール性を足してきました。

エージェントの約束は「DAG を捨て、目標と遷移だけ渡せば LLM が経路を選ぶ」ことでした。しかし実運用では、80% 品質で止まり、残り 20% でフレームワークを逆解析して作り直す、というパターンが繰り返される、と著者は述べています。

3 — エージェントループ
基本パターン

エージェントは次のループで動きます。

  1. LLM が次のステップ（構造化 JSON / ツール呼び出し）を決める
  2. 決定的なコードがツールを実行する
  3. 結果をコンテキストウィンドウに追記する
  4. 完了 intent になるまで繰り返す
initial_event = {"message": "..."}
context = [initial_event]
while True:
  next_step = await llm.determine_next_step(context)
  context.append(next_step)
  if next_step.intent == "done":
    return next_step.final_answer
  result = await execute_step(next_step)
  context.append(result)

4 — なぜ 12 原則か

著者が HumanLayer で見てきた典型的な旅路は次のとおりです。

  1. エージェントを作ると決める
  2. プロダクト設計・UX を固める
  3. フレームワークで素早く 70〜80% まで到達する
  4. 顧客向けには 80% では足りないと気づく
  5. プロンプト・フローを逆解析し、結局ゼロから作り直す
推奨アプローチ: フレームワーク丸ごとではなく、エージェント構築の小さなモジュール概念（構造化ツール呼び出し、コンテキスト設計、人間承認など）を既存プロダクトに組み込む。熟練のソフトウェアエンジニアでも AI 専門でなくてよい、とガイドは述べています。

LLM がさらに賢くなっても、信頼性・スケール・保守性のためのコア技法は残る、という前提で 12 項目が書かれています。

5 — 12 の原則
一覧（原文へのリンク）
#原則要点（要約）
— Brief History of Software DAG・オーケストレータからエージェントループへの文脈
1 Natural Language to Tool Calls 自然言語を構造化ツール呼び出しに変換し、決定的コードで実行する
2 Own your prompts プロンプトをアプリのコードとして所有・版管理する
3 Own your context window メッセージ形式に縛られず、最良の入力を設計する（コンテキストエンジニアリング）
4 Tools are structured outputs ツールは構造化出力の一形態として扱う
5 Unify execution state 実行状態とビジネス状態をコンテキスト上で統一する
6 Launch / Pause / Resume 単純な API で起動・一時停止・再開できるようにする
7 Contact humans with tools 人間への確認もツール呼び出しとしてモデル化する
8 Own your control flow ループ・分岐・エラー処理を自前で持つ
9 Compact errors エラーをコンテキストに収まる形で圧縮して渡す
10 Small, focused agents スコープを小さく保ち、万能エージェントを避ける
11 Trigger from anywhere CLI・Webhook・Cron など入口を増やし、ユーザーの居場所で動かす
12 Stateless reducer エージェントを状態なしリデューサーとして設計する（fold 的な更新）
13* Pre-fetch context（付録） 必要になりそうなコンテキストを先に取りに行く

各原則には img/ 配下の図解（PNG/GIF）が対応しています。README の Visual Nav グリッドからも参照できます。

├── packages/
│   ├── create-12-factor-agent/   # テンプレート生成（npx）
│   └── walkthroughgen/            # ワークショップ用ジェネレータ

└── drafts/                # a2h 仕様など草案

content/ がガイドの正本です。番号付き factor-0N-*.md が各原則の詳細で、重複していた旧ファイル名（factor-N- 単数形）はスタブ程度の行数のみ、とスキャン結果から判断できます。

7 — ツールと学習

create-12-factor-agent

packages/create-12-factor-agent は TypeScript 向けのスターターです。テンプレートには src/agent.ts、cli.ts、server.ts、state.ts などがあり、ワークショップの最終形に近い構成です。README では npx/uvx create-12-factor-agent の議論スレッドも案内されています。

workshops

workshops/2025-05/ などはセクションごとに段階を分けた教材です。CLI とエージェント、計算機ツール、BAML テスト、コンテキストウィンドウ、人間承認、HumanLayer Webhook まで、factor に沿って積み上げます。

walkthroughgen

packages/walkthroughgen は walkthrough YAML から教材コードを生成する CLI です。workshop ディレクトリの walkthrough.yaml と連携します。

技術スタックの例: ガイド本文は TypeScript を多く引用しますが、原則は Python など他言語にも適用可能、と README は明記しています。構造化出力には BAML などの言及もあります。
