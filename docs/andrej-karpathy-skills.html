andrej-karpathy-skills

Karpathy-Inspired Claude Code Guidelines — 単一の CLAUDE.md と Agent Skill で配布される行動規範

1 — 概要
何のためか

このリポジトリは、Andrej Karpathy が指摘した LLM コーディングの癖に対し、4 つの原則でブレーキと方針を与えるための素材をまとめたもの。中核は CLAUDE.md（プロジェクト直下に置く指示）と、Claude Code プラグインとしての Skill（karpathy-guidelines）である。

Think前提を明示
Simple最小のコード
Surgical触る範囲を限定
Goals検証可能なゴール

キーインサイト（README より）: 「LLM は具体的なゴールがあれば、それを満たすまでループするのが得意」— だから手順命令より成功条件と検証を渡す設計に寄せる。

2 — 問題意識（出典の要約）
Karpathy が挙げたパターン

「モデルはユーザーに代わって誤った前提を置き、確認せずに走り続ける。混乱の管理・質問・矛盾の表面化・トレードオフ提示・本来押し返すべきときの押し返しが弱い。」

「コードや API を過剰に複雑にし、抽象を膨らませ、デッドコードを片付けない… 100 行で足りるところを 1000 行の構築物にしてしまう。」

「タスクと直交するコメントやコードまで、十分に理解しないまま副作用として変えてしまうことがある。」

3 — 4 原則
原則と狙い
原則主に抑える失敗
Think Before Coding暗黙の前提、隠れた混乱、トレードオフ未提示
Simplicity First過剰設計、肥大した抽象、不要機能の先回り
Surgical Changes直交するコードへの「ついで修正」、スタイルの押し付け
Goal-Driven Execution曖昧な「なんとかして」、検証なき完了宣言

1. Think Before Coding

実装前に仮説を言語化し、曖昧さは質問で潰す。複数解釈があるときは勝手に一つに決めず並べる。単純な代替があれば指摘する。

2. Simplicity First

依頼範囲を超えた機能・一度きりなのに一般化した抽象・不要なエラーハンドリングを避ける。「シニアが過剰と言うなら削る」というテストを README でも提示。

3. Surgical Changes

隣の未壊コードはリファクタしない。既存スタイルに合わせる。自分の変更で不要になった import 等だけ掃除し、元から死んでいたコードの削除は依頼があるまでしない。

4. Goal-Driven Execution

「バグを直す」→「再現テストを書き、通るまで」など、検証可能な形に言い換える。複数ステップなら ステップ → verify: チェック の短い計画を先に出す。

5 — 導入
Claude Code プラグイン（推奨）

マーケットプレイス登録後、プラグイン名でインストールする流れ（README 準拠）。

/plugin marketplace add forrestchang/andrej-karpathy-skills

/plugin install andrej-karpathy-skills@karpathy-skills

CLAUDE.md をプロジェクトに取り込む

新規プロジェクトでは raw の CLAUDE.md を取得。既存プロジェクトでは追記。

curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md

echo "" >> CLAUDE.md
curl https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md

Skill 本体のエントリは skills/karpathy-guidelines/SKILL.md。エージェント向けの説明文と YAML フロントマター（name / description / license）が定義されている。

6 — 効き目の目安
うまくいっているサイン
  • diff に依頼と無関係な変更が減る
  • 最初から過剰な抽象で書き直す回数が減る
  • 実装より先に確認質問が増える（失敗後ではなく）
  • PR が小さく、ドライブバイリファクタが載りにくい
7 — トレードオフ
速度とのバランス
注意: ガイドラインは慎重さを速度より優先するバイアスを持つ。タイポ修正のような自明な作業では、フル手順までは不要でよい（README の Tradeoff Note と同旨）。

プロジェクト固有ルールは ## Project-Specific Guidelines などで追記し、この 4 原則と合成して使う想定。
