autoresearch

AI エージェントが train.py を編集し、5分固定で訓練して val_bpb を競い、改善だけを残して徹夜実験する。

1 — 概要
何か

autoresearch は、Andrej Karpathy による「AI エージェントに小さな LLM 訓練環境を与え、一晩自律実験させる」リポジトリ。エージェントは train.py を書き換え、5 分訓練して val_bpb（validation bits per byte、低いほど良い）を計測し、改善ならコミットを残し、悪化なら git reset で戻す。人間は program.md でエージェントの指示だけをいじり、Python は触らない。

5 min固定訓練時間
val_bpb唯一の指標
1 file編集は train.py のみ
~12/h実験回数目安

nanochat を単一 GPU・単一ファイルに簡略化した訓練コードで、エージェントがアーキテクチャ・オプティマイザ・ハイパーパラメータを自由にいじれる。朝起きると実験ログと（うまくいけば）より良いモデルが残っている、というコンセプト。

2 — 仕組み
3 つのファイル

リポジトリで「動く」のは次の 3 本。役割がはっきり分かれている。

ファイル役割編集する人
prepare.py定数（MAX_SEQ_LEN, TIME_BUDGET 等）、データ取得・BPE トークナイザ、データローダ・評価。読み取り専用。誰も編集しない
train.pyGPT モデル、オプティマイザ（Muon + AdamW）、訓練ループ。アーキテクチャ・バッチサイズ・ハイパーパラメータすべてここ。AI エージェント
program.mdエージェント向けの指示（セットアップ手順・実験ループ・results.tsv の書き方・「止まるな」など）。人間

訓練は 壁時計で 5 分（起動・コンパイル除く）に固定。指標は val_bpb のみで、vocab に依存しないのでアーキテクチャ変更同士を公平に比較できる。

データ・キャッシュ

uv run prepare.py で ~/.cache/autoresearch/ にデータシャードとトークナイザが用意される。デフォルトは climbmix-400b のシャードと vocab_size=8192 の BPE。prepare.py の定数（MAX_SEQ_LEN, EVAL_TOKENS 等）は変更しない前提。

3 — 実験ループ
エージェントの流れ

program.md に従い、エージェントは専用ブランチ（例: autoresearch/mar5）で次のループを回し続ける。

train.py を変更 → git commit → uv run train.py > run.log → val_bpb / peak_vram を取得 → results.tsv に記録 → 改善なら keep / 悪化なら git reset
  • 最初の 1 本: ベースライン計測のため、何も変えずに uv run train.py を 1 回実行する。
  • results.tsv: タブ区切り。列は commit, val_bpb, memory_gb, status（keep / discard / crash）, description。git にはコミットしない（untracked）。
  • クラッシュ: OOM やバグで落ちた場合は status=crash で記録。 typo など簡単な修正はして再実行、アイデアが根本的にダメならスキップして次へ。
  • NEVER STOP: プログラム上は「人間が止めるまでループ」が前提。寝ているあいだに約 100 実験、という想定。
出力例

訓練終了時に標準出力に出るサマリの例。val_bpb と peak_vram_mb を grep で取り、TSV と比較判定に使う。

---
val_bpb:          0.997900
training_seconds: 300.1
peak_vram_mb:     45060.2
mfu_percent:      39.80
num_params_M:     50.3
depth:            8
4 — クイックスタート
手順

要望: 単一 NVIDIA GPU、Python 3.10+、uv。

# 1. 依存インストール
uv sync

# 2. データ・トークナイザ（初回のみ、数分）
uv run prepare.py

# 3. 訓練 1 本（約 5 分）
uv run train.py

ここまで問題なければ、Claude/Codex 等をこのリポジトリで開き、program.md を読ませて「実験を始めて」と指示すれば自律ループに入る。

プロジェクト構成 
prepare.py      — 定数・データ準備・ランタイム用（編集しない）
train.py        — モデル・オプティマイザ・訓練ループ（エージェントが編集）
program.md      — エージェント指示
pyproject.toml  — 依存（uv）
5 — 設計・プラットフォーム
設計上の選択
  • 編集は 1 ファイルだけ: エージェントは train.py のみ触る。スコープを絞り、diff も追いやすい。
  • 固定時間: 5 分で打ち切りなので、ハードが違っても「そのマシンで 5 分間にどこまでいけたか」で比較できる。代わりに他マシンとの直接比較はしにくい。
  • シンプルさ: 分散訓練なし・設定ファイルもほぼなし。1 GPU、1 ファイル、1 指標。

プラットフォーム

公式は単一 NVIDIA GPU（H100 等でテスト）。CPU / MPS 対応は nanochat 本家やフォークを参照。小さいマシン向けには README に TinyStories データ・vocab 削減・MAX_SEQ_LEN 短縮・DEPTH 下げなどのヒントと、MacOS/Windows 用フォークへのリンクがある。
