Effect

TypeScript 向け 本格的な関数型エフェクトシステム — effect-ts/effect

1 — 概要
What is Effect?

Effect は TypeScript で堅牢なアプリケーションを構築するためのフルスタックフレームワーク。 副作用管理・型安全な並行処理・依存性注入・エラーハンドリングを一つの一貫したプリミティブ Effect<A, E, R> で表現する。 公式サイト と APIリファレンス を参照。

2 — コア型シグネチャ
Effect の型
TypeScript
// Effect<A, E, R> — 3つの型パラメータ
type Effect<A, E, R> = (services: Context<R>) => Promise<Exit<A, E>>

// 簡単な例
const program: Effect.Effect<number, Error, never> = Effect.succeed(42)
const safeDivide = (a: number, b: number): Effect.Effect<number, Error, never> =>
  b === 0
    ? Effect.fail(new Error("Division by zero"))
    : Effect.succeed(a / b)

// 依存サービスを要求する例
interface Database { query: (sql: string) => Effect.Effect<Row[], DbError, never> }
const DbTag = Context.GenericTag<Database>("Database")
const fetchUsers: Effect.Effect<Row[], DbError, Database> =
  Effect.flatMap(DbTag, (db) => db.query("SELECT * FROM users"))
A
Success Value
Effect が正常終了したときに返す値の型。Effect.succeed(42) なら A = number。
E
Error Channel
型安全なエラー。Effect.fail(new Error()) なら E = Error。エラーが発生しない場合は never。
R
Requirements
実行に必要な依存サービスの型（Context<R>）。依存なしの場合は never。

Exit と Cause： Exit<A, E> は実行結果の代数的データ型（Success<A> | Failure<E>）。 Cause<E> はエラーの構造を細かく表現し、予期されたエラー・予期せぬ欠陥・中断・複合エラーを区別できる。
3 — コアモジュール

effect パッケージには 175+ モジュールが含まれる。以下は特に重要な 8 つのコアモジュール。

Effect
副作用・並行処理・エラーハンドリングを管理するコア抽象。map, flatMap, zip, race, retry など豊富な演算子を提供。
Context
型安全な依存性注入。Context.GenericTag でサービスを識別し、直接参照なしに計算を通じてサービスを伝播させる。
Layer
依存関係グラフを管理するシステム。モジュール式で合成可能なリソース確保のしくみ。複雑な DI をレイヤードアーキテクチャで表現。
Fiber
リソースセーフなキャンセルを持つ軽量仮想スレッド。Effect.fork で並行実行、Fiber.join で結果を待機。Effect の並行処理の基盤。
Stream
非同期・イベント駆動のデータ処理抽象。無限ストリーム・バックプレッシャー・リソース管理を型安全に扱える強力な API。
Schedule
リトライ・繰り返しポリシーを合成可能なスケジュールで定義。指数バックオフ・最大試行回数・期限などを組み合わせ可能。
Scope
リソースのライフサイクル管理。取得と解放のペアをスコープに結びつけ、リソースリークを防止。acquireRelease パターン。
Schema
構造化データの定義・検証・変換ライブラリ。型安全なエンコード／デコード。Zod との比較は schema-vs-zod.md を参照。

5 — AI統合
@effect/ai エコシステム

@effect/ai は Effect の型安全なプリミティブを使って複数の AI プロバイダを統一 API で操作できるようにするパッケージ群。 テキスト生成・ツール呼び出し・ストリーミングを抽象化し、プロバイダを差し替えても同じコードが動作する。

OpenAI
@effect/ai-openai
GPT シリーズ・Embeddings・Function Calling を Effect プリミティブでラップ。
Anthropic
@effect/ai-anthropic
Claude シリーズとの統合。ツール呼び出し・ストリーミング対応。
Amazon Bedrock
@effect/ai-amazon-bedrock
AWS Bedrock 経由でマルチモデルを統一インターフェースで利用。
Google AI
@effect/ai-google
Gemini シリーズとの統合。マルチモーダル入力対応。
OpenRouter
@effect/ai-openrouter
OpenRouter 経由で 200+ モデルへアクセス。コスト最適化・フォールバック管理。
@effect/ai（コア）
@effect/ai
プロバイダ共通の抽象インターフェース。AiModel・AiToolkit 等の基礎型を定義。
TypeScript — プロバイダ差し替えの例
import { Effect } from "effect"
import * as Anthropic from "@effect/ai-anthropic"
import * as OpenAI from "@effect/ai-openai"

// ビジネスロジックはプロバイダに依存しない
const summarize = (text: string) =>
  AiModel.generateText({ prompt: `Summarize: ${text}` })

// Layer を差し替えるだけでプロバイダを変更
const withAnthropic = summarize("...").pipe(
  Effect.provide(Anthropic.Claude.layer({ model: "claude-opus-4-5" }))
)
const withOpenAI = summarize("...").pipe(
  Effect.provide(OpenAI.Gpt.layer({ model: "gpt-4o" }))
)
6 — 開発ワークフロー

PR を出す前に以下のコマンドをすべて通す必要がある。pnpm install には pnpm 10.4.0 以上が必要。

01
セットアップ
Fork 後に pnpm install。pnpm codegen でパッケージエントリポイントを再生成（構造変更時）。
型チェック
pnpm check でコンパイルエラーがないことを確認。TS 5.4+ strict モードが対象。
02
03
テスト実行
pnpm test で全ユニットテスト。pnpm test-types で tstyche による型レベルテストも実行。
品質チェック
pnpm circular で循環依存確認。pnpm lint でスタイル検査（自動修正は pnpm lint-fix）。
04
05
Changeset
pnpm changeset で変更を記録。patch（小修正）・minor（新機能）・major（破壊的変更）から選択。
PR 提出先ブランチ
main（バグ修正・小パッチ）・next-minor（新機能）・next-major（破壊的変更）のいずれかに対して PR を出す。
06
コマンド 用途
pnpm install依存パッケージをインストール（pnpm 10.4.0 以上が必要）
pnpm codegenパッケージエントリポイントの再生成
pnpm checkTypeScript コンパイルエラーの確認
pnpm test全ユニットテストの実行
pnpm test-typeststyche による型レベルテストの実行
pnpm circular循環 import の検出
pnpm lintコーディングスタイルの検査
pnpm lint-fixスタイル問題の自動修正
pnpm docgenドキュメントの生成・検証
pnpm changeset変更ログの記録（patch / minor / major）
