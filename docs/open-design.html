Open Design

Local-first design shell for the coding agent you already use. Claude Design の「成果物ファースト」な体験を、既存CLI・ファイルベースSkills・Markdown Design Systemsで開く。

1 — 概要
何が新しいか

Open Design は、Claude Code、Codex CLI、Cursor Agent、Gemini CLI、OpenCode、Qwen Code、GitHub Copilot CLI など、利用者の手元にあるコーディングエージェントを「デザインエンジン」として起動するWebアプリ。プロダクト自体はモデルもエージェントループも抱え込まず、SKILL.md と DESIGN.md を合成して、HTMLプロトタイプ・スライドデッキ・ドキュメントテンプレートを作らせる。

BYOKat every layer

核は「作る前に質問フォームで前提を固める」「Skillが成果物の型を決める」「Design Systemが見た目を決める」「daemonがローカルCLIを実行する」の4点。AIが自由演技でデザインするのではなく、ファイル化された制約とチェックリストで出力のブレを抑える設計になっている。

2 — 生成ループ
ユーザー操作から成果物まで
01
Brief を受ける

ユーザーがSkill、Design System、短い依頼文を選ぶ。新規依頼では質問フォームを先に出し、出力面・対象読者・トーン・規模を確定する。

02
Prompt stack を合成

基礎プロンプト、発見質問、アクティブな DESIGN.md、アクティブな SKILL.md、必要なテンプレート参照を一つの実行文脈に束ねる。

03
ローカルCLIを起動

daemonが検出済みエージェントをプロジェクト用の作業ディレクトリで起動し、標準出力をSSEとしてUIへ流す。CLIがなければBYOK API経路へ落ちる。

04
Artifact をプレビュー

モデル出力から <artifact> を取り出し、サンドボックスiframeに表示する。保存・ZIP・PDF・PPTXなどの出力はSkillの型に沿って扱う。

Prompt stack

Open Designでは「プロンプトスタックがプロダクト」。コード上では、発見質問と設計哲学を最優先に置き、その後ろに公式デザイナー憲章、Design System、Skill、デッキ用フレームワークを重ねる。

DISCOVERY directives
  + identity and workflow charter
  + active DESIGN.md
  + active SKILL.md
  + project metadata
  + skill side files
  + deck framework directive
設計判断: Open Designはエージェント本体を所有しない。既存CLIが得意なtool loopや権限管理を使い、Open DesignはSkill、Design System、プレビュー、保存、エクスポートの「統合シェル」に徹する。
3 — 構成

Web App

Next.js 16 + React のUI層。チャット、Skill / Design System選択、artifact tree、sandboxed iframeプレビュー、エクスポート導線を持つ。

Local daemon

Expressベースの特権側プロセス。エージェント検出、Skill登録、Design System解決、チャットSSE、ファイル保存、エクスポートを担当する。

Agent adapters

各CLIの検出と起動引数を吸収する薄い層。構造化ストリームを持つCLIはtyped events、その他はline-buffered出力で扱う。

Contracts

Webとdaemonが共有する型とAPI契約。プロジェクト、chat、files、artifacts、registryなどの境界をTypeScriptで揃える。

Skills

成果物タイプごとの手順書。SKILL.md、テンプレート、参照ファイル、チェックリストを1フォルダに閉じ込める。

Design Systems

ブランドや視覚言語のMarkdown仕様。色、タイポ、余白、コンポーネント、声のトーンを9セクションで記述する。

4 — Skills と Design Systems
Skill は成果物の型

Skillは「何を作るか」と「どの順番で作るか」を定義する。ランディングページ、ダッシュボード、価格ページ、モバイル画面、横スワイプdeck、週報、OKR、runbookなど、成果物ごとの作法をフォルダで増やせる。

web-prototype saas-landing dashboard mobile-app guizang-ppt pm-spec

Design System は見た目の言語

Design Systemは「どう見えるか」を決める。Neutral ModernとWarm Editorialのスターターに加え、Linear、Stripe、Vercel、Airbnb、Tesla、Notion、Apple、Cursor、Supabaseなどのプロダクト風システムが同梱されている。

軸役割ユーザー体験への効き方

5 — 4つのモード
Mode作るもの代表的な流れ
Prototype単一画面または短いUIフローSkill選択 → brief → iframe preview → コメントやチャットで refinement → HTML / PDF / ZIP
Deck横スワイプのHTMLスライドやPPTX向け中間成果物枚数・テーマ・アウトラインを決め、Skillのdeck frameworkに内容を流し込む
Template完成済みテンプレートの内容差し替え構造や見た目は固定し、ユーザーのコピーやブランド要素だけを埋める高速モード
Design System再利用可能な DESIGN.mdスクリーンショット、ブランドガイド、URL、テキスト brief から9セクションの設計言語を生成する
組み合わせの考え方: Design Systemを先に作ると、Prototype、Deck、Templateが同じ視覚言語を共有できる。つまりOpen Designの本質は「成果物モード × Skill × Design System」の直積にある。
6 — 品質ゲート
AI-slop を減らす仕掛け
  • Turn 1ではいきなりHTMLを書かず、質問フォームで未確定の設計判断を固定する。
  • ブランド未指定なら、5つの視覚方向から選ばせ、パレットとフォントを deterministic に束ねる。
  • Skillが持つテンプレートや参照ファイルを事前に読ませ、毎回ゼロからCSSを発明させない。
  • 出力前にP0/P1/P2チェックリストと5次元自己批評を走らせる。
セキュリティ境界
  • プレビューは <iframe sandbox="allow-scripts"> でホストUIから隔離する。
  • secretsは利用者のdaemonまたはブラウザ側に留まり、Open Design独自のホストサーバーへ預けない前提。
  • Skillは強力な指示ファイルなので、インストール時に信頼境界を意識する必要がある。
  • daemonを外へ公開する場合は、SSEのバッファリングや認証・トンネル設定が運用上の論点になる。
