MarkItDown

Various files → Markdown（LLM・テキスト解析パイプライン向け）

1 — 概要
位置づけ

MarkItDown は Microsoft が公開している軽量な Python ユーティリティで、PDF・Office・HTML・画像・音声など多様な入力を Markdown に揃えて出力する。人間向けの高忠実度変換よりも、見出し・リスト・表・リンクなどの構造を残しつつ LLM が扱いやすいテキストを優先する設計（microsoft/markitdown）。

3.10+Python
CLIstdin / -o
APIMarkItDown クラス
MCPmarkitdown-mcp

主な入力形式（README より）

  • PDF / PowerPoint / Word / Excel
  • 画像（EXIF・OCR 系オプション）/ 音声（メタデータ・文字起こし）
  • HTML・CSV / JSON / XML などテキスト系・ZIP（中身を列挙）
  • YouTube URL・EPub など（詳細は 公式 README）

textract に近い「なんでもテキスト化」だが、出力は Markdown 中心。GPT 系モデルが Markdown を強く理解しているという前提に沿い、トークン効率も意識されている。

なぜ Markdown か

プレーンテキストに近くトークン効率がよく、見出し・リスト・表・リンクで構造を表現できる。多くの LLM が Markdown を前提に学習・応答しており、RAG や後段のテキスト解析に載せやすい、という README の位置づけ。

Breaking（0.0.1 → 0.1.0 付近）: 依存は optional グループに整理（後述の pip install 'markitdown[all]' 推奨）。convert_stream() は バイナリの file-like のみ。DocumentConverter はパスではなく ストリームから読む インターフェイスに変更され、プラグイン実装者は要確認。
2 — 対応形式
入力

コアパッケージは PDF・PowerPoint・Word・Excel・画像（EXIF / OCR）・音声（EXIF / 文字起こし）・HTML・CSV/JSON/XML・ZIP（中身を列挙）・YouTube URL・EPub などを Markdown へ変換する。出力は LLM やテキスト解析パイプライン向けで、人間向けの高忠実度変換を最優先する設計ではない。

カテゴリ例
ドキュメントPDF, DOCX, PPTX, XLSX/XLS, EPub
ウェブ・テキストHTML, CSV, JSON, XML, RSS, Wikipedia
メディア画像（メタデータ・OCR）, 音声（文字起こし）, YouTube
その他ZIP（中身を順次変換）, Outlook MSG, Bing SERP
3 — インストール
前提・導入

Python 3.10 以上。仮想環境の利用を推奨。

python -m venv .venv
source .venv/bin/activate
pip install 'markitdown[all]'

ソースから入れる場合:

git clone https://github.com/microsoft/markitdown.git
cd markitdown
pip install -e 'packages/markitdown[all]'

Docker

docker build -t markitdown:latest .
docker run --rm -i markitdown:latest < ~/your-file.pdf > output.md
4 — 変換パイプライン
ざっくり流れ

MarkItDown.convert() はローカルパス・URL・requests.Response・バイナリストリームを受け取り、内部で Magika 等による内容推定と StreamInfo の候補リストを作り、登録済み DocumentConverter を優先度順に試す。

入力 → StreamInfo 推定 → コンバータ選択 → Markdown 文字列
  • 優先度: 具体的な拡張子・MIME 向けコンバータ（PRIORITY_SPECIFIC_FILE_FORMAT）が汎用（PlainTextConverter など）より先に試されるよう、後から登録したものほど優先というスタック順（コメントより: より専用なコンバータをリストの下側に）。
  • Azure Document Intelligence: エンドポイントを渡した場合のみ DocumentIntelligenceConverter が追加登録され、PDF 等の処理に使える。
  • プラグイン: entry_points(group="markitdown.plugin") で読み込み、enable_plugins=True で各プラグインがコンバータを登録。
HTTP 取得時のヘッダ

デフォルトの requests.Session は Accept: text/markdown, text/html;q=0.9, ... を付与し、サーバが Markdown を返せる場合にそれを優先する（README の Cloudflare 記事リンク参照）。

5 — CLI・Python API
CLI 
markitdown path-to-file.pdf > document.md
markitdown path-to-file.pdf -o document.md
cat path-to-file.pdf | markitdown

Docker（README より）

docker build -t markitdown:latest .
docker run --rm -i markitdown:latest < ~/your-file.pdf > output.md

プラグイン一覧: markitdown --list-plugins。有効化: markitdown --use-plugins path-to-file.pdf。

Python（最小） 
from markitdown import MarkItDown

md = MarkItDown(enable_plugins=False)
result = md.convert("test.xlsx")
print(result.text_content)

LLM で画像説明（例: 画像・pptx）

from markitdown import MarkItDown
from openai import OpenAI

client = OpenAI()
md = MarkItDown(llm_client=client, llm_model="gpt-4o", llm_prompt="optional custom prompt")
result = md.convert("example.jpg")
print(result.text_content)
6 — Azure 連携
Document Intelligence

スキャン PDF や複雑なレイアウト向け。CLI では -d -e <endpoint>、Python では docintel_endpoint を渡す。

from markitdown import MarkItDown

md = MarkItDown(docintel_endpoint="<document_intelligence_endpoint>")
result = md.convert("test.pdf")
print(result.text_content)
Content Understanding

音声・動画、構造化フィールド（YAML front matter）、カスタムアナライザーなど、ビルトインや Document Intelligence では足りないケース向け。各 convert() は課金 API 呼び出しになり得る。

能力ビルトインDoc IntelContent Understanding
ドキュメント変換オフライン・形式別クラウドレイアウトクラウド多モーダル
構造化フィールドなし本統合では未露出YAML front matter
音声・動画基本音声のみ非対応対応
コストローカル計算Azure 従量Azure 従量
 
from markitdown import MarkItDown

md = MarkItDown(cu_endpoint="<content_understanding_endpoint>")
result = md.convert("report.pdf")   # documents
result = md.convert("meeting.mp4")  # video
print(result.markdown)

Extra: pip install 'markitdown[az-content-understanding]'。CLI: markitdown file.pdf --use-cu --cu-endpoint "<endpoint>"

9 — セキュリティ
入力の扱い

MarkItDown は open() や requests.get() と同様、プロセスがアクセスできるリソースに I/O する。信頼できない入力をそのまま渡さないこと。

  • サーバー側利用ではパス・URI スキーム・ネットワーク先を制限し、入力を検証する。
  • 用途に合わせて最も狭い API を選ぶ: ローカルファイルのみなら convert_local()、HTTP は自前で requests.get() して convert_response()、最大制御なら convert_stream()。
  • convert() はパス・URL・ストリームを広く受け付けるため、制御が必要な環境では避ける。
