# PR テンプレート

> **原則**: レビュアーは忙しい。「読みたい」と思っていない前提で、読んでもらう努力をする。

## 本文テンプレート

```markdown
## なぜこの変更が必要か（背景）
<!-- レビュアーが最初に知りたいのは「WHY」 -->

## 何を変えたか（概要）
<!-- 1-2文で要約 -->

## 変更内容
<!-- 主な変更点を箇条書き -->
-

## レビューのポイント
<!-- レビュアーの時間を尊重する。どこを重点的に見てほしいか -->
- 特に見てほしい箇所:
- 判断を仰ぎたい点:

## 確認済み事項
<!-- 自分でどこまで検証したか。レビュアーの安心材料 -->
- [ ] ローカルで動作確認
- [ ] lint/build通過
- [ ] 関連テスト通過

## 関連
<!-- 依存PRがある場合のみ記載 -->
Depends on #<PR番号>
```

## タイトル形式

| 形式 | パターン | 例 | 条件 |
|------|---------|---|------|
| **コードネーム付き** | `[コードネーム] PR-XX: 内容` | `[login] PR-00: Design Doc作成` | Design Docsあり |
| **パッケージ付き** | `PR-XX: パッケージ: 内容` | `PR-00: common: 型定義の追加` | Design Docsあり |
| **シンプル** | `PR-XX: 内容` | `PR-00: 初期実装` | Design Docsあり |
| **プレーン** | `内容` | `GitButler CLIスキルの追加` | Design Docsなし |

## タイトル決定ロジック

1. **Design Docsの有無を判断**: ブランチ名が `feature/{codename}/{NN}-*` パターンに該当するか
   - **該当しない** → **プレーン形式**を使用（コミットメッセージから内容を推測）
   - **該当する** → 次のステップへ
2. **ブランチ名パターンから推測**:
   - `feature/{codename}/{NN}-*` → `[{codename}] PR-{NN}: {内容}`
   - `fix/{codename}/{NN}-*` → `[{codename}] PR-{NN}: {内容}`
   - `hotfix/*` → シンプル形式を使用
3. **コードネームが不明な場合**: 最新のコミットメッセージを参考にシンプル形式を使用
4. **判断できない場合**: ユーザーに確認

## 例: 本文の実際の記述

### 良い例

```markdown
## なぜこの変更が必要か（背景）
ログイン機能の要件が確定したため、Design Docを作成して実装方針を明確にする。

## 何を変えたか（概要）
ログイン機能のDesign Docを新規作成。

## 変更内容
- docs/design-docs/login.md を新規作成
- 認証フローの設計を記載
- セキュリティ要件を明記

## レビューのポイント
- 特に見てほしい箇所: 認証フローの設計が妥当か
- 判断を仰ぎたい点: セッション管理の方式（JWT vs Cookie）

## 確認済み事項
- [x] Markdownの構文チェック
- [x] リンク切れ確認
```

### 悪い例

```markdown
## 概要
ログイン機能のDesign Doc作成しました。
```

**問題点**:
- WHYがない（なぜ今この変更が必要か不明）
- レビューのポイントがない（何を見ればいいか不明）
- 確認済み事項がない（どこまで検証済みか不明）
