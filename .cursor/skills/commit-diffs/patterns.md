# 分割パターンとサンプル

> **目的**: よくある変更パターンに対する最適な分割方法を提供する

## 分割パターン一覧

| シナリオ | 推奨分割 | 理由 |
|----------|----------|------|
| フォーマット＋実装 | style → feat/fix | フォーマットノイズを除去してから本質的変更 |
| リファクタ＋機能追加 | refactor → feat | 基盤整備してから機能追加 |
| 依存更新＋コード変更 | deps → feat/fix | 依存変更を分離して影響範囲を明確化 |
| スキーマ変更＋コード | migration → adapt | 段階的に移行 |
| テスト追加＋実装 | 同時OK（TDD的に1コミット可） | 実装とテストは論理的に1単位 |
| 複数ファイルの同一修正 | 1コミットでOK | 同じ目的なら分割不要 |

## Good / Bad Examples

### ✅ Good: 適切な分割

**Before（1つの大きな差分）**:
- ユーザー一覧画面にページネーション追加
- 同時にリファクタ（関数分割）
- ついでにフォーマット修正

**After（3つのコミット）**:
```
1. style: ユーザー一覧画面のコードフォーマットを統一する
2. refactor: ユーザー一覧画面のデータ取得ロジックを分離する
3. feat: ユーザー一覧画面にページネーションを追加する
```

**分割の理由**:
- style: フォーマット変更は差分ノイズ → 先に処理
- refactor: 構造改善は振る舞い不変 → 機能追加前に実施
- feat: 本質的な機能追加を最後に

---

### ✅ Good: 依存関係を考慮した分割

**Before（1つの大きな差分）**:
- 新しい型定義 `UserDTO` を追加
- その型を使う関数を複数追加
- テストも追加

**After（2つのコミット）**:
```
1. feat: UserDTO型定義を追加する
2. feat: UserDTO型を使用するユーザー取得機能を実装する
```

**分割の理由**:
- 型定義が先 → 使用箇所が後（依存関係の順序）
- テストは実装と同時（TDD的に1単位）

---

### ❌ Bad: 不適切な分割

```
# NG: 複数の目的が混在
変更

# NG: 意図が不明
修正

# NG: 粒度が大きすぎ
機能追加とバグ修正とリファクタ

# NG: 過度な分割
feat: 関数を追加
feat: 関数を呼び出す
feat: 戻り値を変数に代入
```

**問題点**:
- 「変更」「修正」: 何をなぜ変えたか不明
- 混在: 巻き戻し時に意図しない変更も戻る
- 過度な分割: 論理的に1単位のものを分けすぎ

---

## シナリオ別ガイド

### シナリオ1: バグ修正 + フォーマット混在

**状況**: バグを修正したが、エディタが自動フォーマットを適用した

**🅶 Git 環境での対処**:
```bash
# 1. フォーマット変更のみをステージ
git add -p  # フォーマット変更のhunkのみ選択

# 2. styleコミット
git commit -m "style: コードフォーマットを統一する"

# 3. バグ修正をステージ
git add -p  # 残りの変更を選択

# 4. fixコミット
git commit -m "fix: ユーザー名が空の場合のエラーを修正する"
```

**🅱️ GitButler 環境での対処**:
```bash
# 1. but status -f でファイルのCLI IDを確認
but status -f
# 例: h0 = フォーマット変更のファイル, i0 = バグ修正のファイル

# 2. フォーマット変更ファイルのみコミット
but commit -p h0 -m "style: コードフォーマットを統一する" <branch>

# 3. バグ修正ファイルのみコミット
but commit -p i0 -m "fix: ユーザー名が空の場合のエラーを修正する" <branch>
```

> **GitButler Tips**: 同一ファイル内にフォーマットとバグ修正が混在する場合、
> `but commit empty <branch>` で空コミットを挿入し、`but rub` で変更を振り分ける。

### シナリオ2: 機能追加 + リファクタ混在

**状況**: 機能追加のために既存コードをリファクタした

**対処**（両環境共通の方針）:
```
1. refactor: 既存の処理を共通関数に抽出する
   - 振る舞い変更なし
   - テストが通ることを確認

2. feat: 新機能を追加する
   - 共通関数を利用
   - 新しいテストを追加
```

**🅱️ GitButler 環境での追加テクニック**:
```bash
# リファクタ用と機能追加用のファイルを別々にコミット
but commit -p <refactor-file-ids> -m "refactor: 既存の処理を共通関数に抽出する" <branch>
but commit -p <feat-file-ids> -m "feat: 新機能を追加する" <branch>

# もしくは、既にコミット済みの場合は rub で再編成
but rub <file-id> <target-commit-id>  # 変更を適切なコミットに移動
```

### シナリオ3: 複数パッケージへの変更

**状況**: monorepoで複数パッケージに跨る変更

**対処**（両環境共通の方針）:
```
# パッケージ間の依存関係を確認
1. common パッケージの変更を先に
2. それを使用する各パッケージの変更を後に

# 例
1. feat(common): 共通ユーティリティを追加する
2. feat(frontend): 共通ユーティリティを使用してUIを実装する
3. feat(backend): 共通ユーティリティを使用してAPIを実装する
```

**🅱️ GitButler 環境での追加テクニック**:
```bash
# パッケージごとにファイルIDで分割コミット
but status -f  # パッケージごとのファイルIDを確認
but commit -p <common-file-ids> -m "feat(common): 共通ユーティリティを追加する" <branch>
but commit -p <frontend-file-ids> -m "feat(frontend): 共通ユーティリティを使用してUIを実装する" <branch>
but commit -p <backend-file-ids> -m "feat(backend): 共通ユーティリティを使用してAPIを実装する" <branch>

# 独立した変更は並列ブランチに分けることも可能
but branch new feat-common
but branch new feat-frontend
but rub <file-id> <branch-id>  # ファイルを適切なブランチに割当
```

## コミットメッセージのルール

| 要素 | ルール |
|------|--------|
| **件名** | 日本語、命令形、〜50文字、末尾ピリオドなし |
| **本文** | 「何を」「なぜ」を説明 |
| **カテゴリ** | feat/fix/refactor/style/deps/docs/test/chore |

### メッセージ形式

```
[カテゴリ]: 概要（〜50文字）

何を変えたか:
- 変更点1
- 変更点2

なぜ変えたか:
- 理由や背景
```

### 良いメッセージ例

```
feat: ユーザー一覧にページネーションを追加する

何を変えたか:
- UserListコンポーネントにページネーションUIを追加
- useUsersフックにページパラメータを追加

なぜ変えたか:
- 1000件超のユーザーで表示が遅くなる問題を解決
```

### 悪いメッセージ例

```
# NG: 何をなぜが不明
修正

# NG: 粒度が不適切
ページネーションを追加してフォーマットを修正してバグも直した
```
