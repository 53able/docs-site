## Context Engineering 視点：なぜ抽出が重要か

Context Engineering Survey (§4.2) が示す通り、**Structured Information Integration** の核心は
「生の情報をそのまま渡すのでなく、エージェントが処理しやすい構造に変換すること」。

```
Raw Source  →  [Structured Extraction]  →  Visual Components
 (冗長・散漫)          (圧縮・分類)           (目的に最適化)
```

> c_know（外部知識）は圧縮されてから c_instr と結合される。
> 生のソースをそのままコンテキストに置くと "Lost in the Middle" 問題（§4.3）が発生する。

## 情報タイプの分類

ソースを読んだら、各パラグラフ・セクションを以下の 6 タイプに分類する。

| タイプ | 識別シグナル | 例 |
|--------|------------|-----|
| **概念説明** | "とは何か" "目的" "なぜ" | README の Overview セクション |
| **手順** | 番号付きリスト, "Step N", "Run X" | SKILL.md の Procedure |
| **設定値/オプション** | key-value 形式, デフォルト値表 | Config options, CLI flags |
| **コード例** | コードブロック, コマンド | スクリプト呼び出し例 |
| **比較** | "vs", "better/worse", good/bad 並列 | BAD/GOOD examples |
| **制約/注意** | "注意", "禁止", "Don't use for", "Error" | Error Handling, Anti-patterns |

## コンテキスト圧縮の手順

読んだファイルを内部表現として圧縮する（コンテキスト汚染を防ぐ）。

### ステップ 1: KSI（Key Structural Items）の特定

各ファイルから最大 7 項目を抽出する:

```
1. 目的（1 文）
2. 主要機能・手順（箇条書き 3-5 項目）
3. 重要な制約（箇条書き 1-3 項目）
4. 典型的な使用例（1-2 個）
5. 関連ファイル・依存（参照元のみ）
```

### ステップ 2: セクション骨格の作成

KSI を基に visual セクションの骨格を作る:

```
セクション 1: 概要（必須）
  → 目的 + KPI カード（定量値があれば）
  
セクション 2-N: 機能・手順
  → 情報タイプに対応するコンポーネントを選ぶ（spell-ui-map.md 参照）

最終セクション: 参考リンク（必須、ソースが外部リンクを持つ場合）
```

### ステップ 3: 圧縮率チェック

- 元ソースの行数に対して、視覚化するコンテンツは **20-40% 相当** が目安
- 全行を網羅しようとしない。**タイプ別に 1-2 個の代表例** を使う
- 「読めばわかる」内容は省略し、「見てわかる」構造に集中する

## ソースタイプ別の抽出パターン

### Pattern A: agentskills.io 形式スキル（SKILL.md + refs/ + assets/）

```
読む順: SKILL.md → 1 つの reference → 1 つの asset
抽出: frontmatter の name/description → 手順セクション → Error Handling
視覚化: ディレクトリ構造 + 手順フロー + チェックリスト
```

### Pattern B: ライブラリ/ツール（README.md 中心）

```
読む順: README → package.json or pyproject.toml → 1 つのコア実装ファイル
抽出: Features / Installation / Usage / API
視覚化: KPI カード（バッジ） + インストールコード + API テーブル
```

### Pattern C: ガイドライン/ベストプラクティス（章立て文書）

```
読む順: 目次 → 各章の先頭パラグラフのみ（実装コードは不要）
抽出: 原則リスト + good/bad 例 + チェックリスト
視覚化: 原則カードグリッド + バッジ付き比較テーブル + チェックリスト
```

### Pattern D: CLI ツール（コマンド中心）

```
読む順: README → --help 出力またはコマンド一覧ファイル
抽出: インストール方法 + コマンド一覧 + 主要オプション
視覚化: コードブロック（インストール） + コマンドテーブル + オプションテーブル
```

## 自己精緻化チェック（Self-Refinement §4.2）

抽出が完了したら以下を確認する:

- [ ] 概念説明が 1 セクションに集約されているか（散在は認知負荷を上げる）
- [ ] 手順がステップ番号と対応しているか（ランダムな箇条書きは禁止）
- [ ] 比較情報が bad/good 対比テーブルになっているか（散文で書かない）
- [ ] 制約・注意が callout または別カードに分離されているか（本文に埋め込まない）
- [ ] コード例が `ve-code-block` 用に整形されているか（空白・改行の確認）
