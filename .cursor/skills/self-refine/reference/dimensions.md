# Multi-Aspect 評価次元

> Self-Refine の Evaluate ステップで使用する 5 つの評価次元の詳細定義。
> 各次元は独立して評価し、NG の場合は具体的なフィードバックを記述する。

## 1. 目的整合性（Purpose Alignment）

**問い**: このプランでユーザーの課題が解決するか？

### チェックポイント

- ユーザーが解決したい問題は何か、正しく言語化されているか
- プランの出力がその問題の解決になっているか
- 「手段の目的化」が起きていないか（やることが目的になっていないか）
- 部分的な解決で終わっていないか

### NG例

```
ユーザーの課題: 「ページの読み込みが遅い」
プラン: 「コードをリファクタリングする」
→ NG: リファクタリングは手段であり、パフォーマンス改善に直結するとは限らない
```

---

## 2. 影響範囲（Impact Scope）

**問い**: 見落としている影響範囲はないか？

### チェックポイント

- 変更するファイル以外に影響を受けるファイルはないか
- 他のパッケージへの影響はないか
- テストコードへの影響はないか
- 型定義やインターフェースの変更による連鎖的影響はないか
- データベースやAPIの変更による後方互換性の問題はないか

### 調査方法

```
影響を見つけるための問い:
├─ この変更の「出力」を使っている箇所は？
├─ この変更の「入力」を提供している箇所は？
├─ 同じインターフェースを実装している他の箇所は？
└─ このモジュールに依存しているテストは？
```

---

## 3. 前提妥当性（Premise Validity）

**問い**: 依拠している前提は正しいか？

### チェックポイント

- プランが依拠している技術的前提は検証済みか
- 「〜のはず」「〜だろう」という推測に基づいていないか
- 暗黙の前提が存在しないか
- 前提が変わった場合にプランが破綻しないか

### よくある危険な前提

| 前提 | リスク |
|------|--------|
| 「この関数は常にXを返す」 | エッジケースで異なる値を返す可能性 |
| 「この処理は高速」 | データ量増加で遅くなる可能性 |
| 「他に影響はない」 | 見落としの可能性 |
| 「仕様はこうなっている」 | 実装が仕様と乖離している可能性 |

---

## 4. 実行可能性（Executability）

**問い**: 具体的に実行できる粒度か？手順に曖昧さはないか？

### チェックポイント

- 各ステップが具体的な操作に落とし込めるか
- 「適切に処理する」のような曖昧な記述がないか
- 変更対象のファイルパスが特定されているか
- 成功基準（テスト、lint、build）が明確か
- 必要な情報が揃っているか、不足していないか

### NG例

```
プランのステップ: 「エラーハンドリングを改善する」
→ NG: どのエラーを、どのように、どこで処理するかが不明
→ 改善: 「fetchData関数のcatchブロックで、NetworkErrorの場合はリトライ、
         ValidationErrorの場合はユーザーにフィードバックを返す」
```

---

## 5. 認知負荷（Cognitive Load）

**問い**: より単純な方法はないか？不要な複雑さを持ち込んでいないか？

### チェックポイント

- 同じ結果を、より少ないステップで達成できないか
- 新しい概念や抽象化を導入しすぎていないか
- 既存のパターンや慣習に従っているか
- 変更量は最小限か（問題に直接関係のない変更を含んでいないか）
- レビュアーが理解しやすいか

### 簡素化の問い

```
簡素化チェック:
├─ このステップは本当に必要か？（削除できないか）
├─ 2つのステップを1つにまとめられないか？
├─ 既存のユーティリティや関数で代替できないか？
└─ この抽象化は本当に必要か？（YAGNI原則）
```
