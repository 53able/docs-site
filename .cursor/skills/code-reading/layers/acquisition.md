# Layer 1: 取得 (Acquisition)

> どの情報を、どの順序で読むか

## 1-1. ゴール定義（最初に必ず行う）

**トリガー**: コードリーディング開始時

```
Q: 何を達成したいか？
├─ バグ修正      → 問題箇所の特定と原因理解
├─ 機能追加      → 拡張ポイントと依存関係の把握
├─ レビュー      → 設計意図と実装品質の評価
├─ 学習          → パターンと設計思想の吸収
└─ 調査          → 特定の振る舞いの確認
```

**出力形式**: 1文でゴールを明文化する

```
例: 「認証フローでトークン検証がどこで行われるかを特定する」
```

## 1-2. 境界設定（スコープの限定）

**原則**: 最小集合から始め、必要に応じて拡張する

```
Step 1: エントリーポイントを1つ特定
Step 2: 直接依存のみを対象範囲に含める
Step 3: 必要になったら拡張（前倒しで広げない）
```

### スコープ拡張の判断基準

```
IF 以下の場合のみ拡張:
├─ 現在の範囲では目的を達成できない
├─ 依存先の理解が必須
└─ エラーの原因が外部にある可能性が高い

ELSE:
└─ 現在の範囲内で完結させる
```

## 1-3. 情報取得の優先順位

**読む順序**（上から順に、必要な分だけ）:

| 優先度 | 対象 | 目的 | 読む量 |
|--------|------|------|--------|
| 1 | ファイル名・ディレクトリ構造 | 責務の推測 | 全体俯瞰 |
| 2 | 型定義・インターフェース | 契約の把握 | 関連部分のみ |
| 3 | 公開API（export）の関数シグネチャ | 入出力の確認 | シグネチャのみ |
| 4 | ドキュメント・コメント | 意図の理解 | 存在すれば |
| 5 | テストコード | 期待される振る舞いの確認 | 関連テストのみ |
| 6 | 実装の詳細 | 「なぜ動くか」のピンポイント確認 | **最小限** |

### 各優先度での確認ポイント

**優先度1: ファイル構造**
```bash
# ディレクトリ構造から責務を推測
ls -la src/
# または
tree src/ -L 2
```

**優先度2: 型定義**
```typescript
// 型定義から契約を把握
export interface UserService {
  findById(id: string): Promise<User | null>;
  // ↑ 入力: string、出力: User | null、非同期
}
```

**優先度3: 公開API**
```typescript
// シグネチャから入出力を確認
export const validateToken = (token: string): ValidationResult => { ... };
// ↑ 実装詳細は後回し
```

## アンチパターン

| パターン | 問題 | 対策 |
|----------|------|------|
| 優先度6から始める | 木を見て森を見ず | インターフェイスから |
| 全ファイルを読もうとする | 時間切れ、目的喪失 | 境界設定を先に |
| 「念のため」で範囲拡張 | 無限散策 | 拡張は証拠に基づいて |
